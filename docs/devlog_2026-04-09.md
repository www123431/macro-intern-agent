## 一、Alpha Memory 学习质量强化

### 1.1 随机抽检 8%（engine/agent.py）

服务模块：Tab4 Quant Audit · 红队审计节点

当前红队只拦截触发硬性规则的分析（p-hacking、特征稀疏、否决词），对于"逻辑看似合理但结论存在系统性偏差"的情况不会介入。随机抽检强制一部分通过审计的分析也进入反思节点，防止模型在某类资产或情景下长期形成"自我验证"的惯性。

- `is_robust=True` 且随机数 < 0.08 时，强制覆盖为 `is_robust=False`
- 触发标记：`🔀 [随机抽检·强制深度审计]`
- 作用：实时审计质量的安全阀

### 1.2 反向证据门槛 ≥10 条（engine/memory.py）

服务模块：Alpha Memory (Tab6) · Meta-Agent

Meta-Agent 归纳的偏差规律本身可能带有确认偏误，容易将短期市场噪声误判为结构性规律。达到阈值后，每条规律在写入 LearningLog 前附带反向证据提示，促使后续 agent 调用时主动寻找反例。

- 阈值：已验证决策 ≥ 10 条
- 追加内容：`⚠ 反向证据提示：请在应用此规律前，主动寻找反例……`
- 作用：历史学习偏差的纠偏机制

### 1.3 统计门槛强化（engine/memory.py）

| 参数 | 旧值 | 新值 | 原因 |
|------|------|------|------|
| `min_samples` | 5 | 25 | 5条数据统计上极不稳定 |
| cell 内部最低样本 | ≥3 | ≥5 | 减少噪声规律触发 |
| 准确率判断依据 | 均值 avg | CI 下界 avg − 1.96×SE | 样本少时 SE 大，下界低，规律更难触发；样本充足时结论才稳定 |

偏差和优势判断均改用 95% 置信区间下界，显示内容同步更新为 `95%CI下界=xx%`。

### 1.4 Pattern 失效机制重设计（engine/memory.py）

**原设计（时间衰减）的问题**：对所有 pattern 一视同仁地按90天半衰期衰减，但金融规律的生命周期取决于类型而非时间。长期结构性规律会被错误淡化，短期周期性规律反而衰减得不够快。

**新设计（证据失效，三层）**：

**第一层 — Regime 门控（替代衰减）**
pattern 不衰减，但只在 `macro_regime` 匹配当前环境时注入 prompt。Regime 切换后 pattern 自动沉默，Regime 回归后自动复活。

**第二层 — 数据反驳（替代硬过期）**
每次 meta-agent 运行时，用最新数据重新计算每条 `bias`/`strength` pattern 对应 cell 的准确率和 CI 下界：
- `bias` pattern：准确率回升至 `MIN_ACCEPTABLE` 以上 → `applied=True`（已解决）
- `strength` pattern：准确率下降至 `EXCELLENT` 以下 → `applied=True`（已失效）
- 失效原因写入 description，完整保留历史

**第三层 — Regime 休眠检测（有限度保留时间概念）**
某 regime 在过去 180 天内未出现于已验证决策 → 对应 pattern 标记 `dormant=True`（沉默但不删除）。该 regime 重新出现时自动复位。

| 维度 | 旧实现 | 新实现 |
|------|--------|--------|
| 失效依据 | 时间（90天半衰期） | 新数据反驳 |
| 长期有效规律 | 被错误淡化 | 持续保留 |
| 已修正的偏差 | 等待自然过期 | 立即归档 |
| Regime 切换后 | 仍然注入 | 自动沉默 |
| 稀有 Regime | 永久保留并注入 | 休眠，不注入 |

**代码变更**：
- `LearningLog` 模型：移除 `expires_at`，新增 `dormant` 布尔字段
- `run_meta_agent_analysis`：新增步骤5a（证据反驳）、步骤5b（Regime 休眠检测）
- `get_learning_patterns`：过滤 `dormant=False`，NULL 旧行视为非休眠
- Admin 面板：新增休眠 pattern 计数提示

**本质目的**：保证注入 prompt 的规律是有效的，不是过期的，提高 agent 决策的信息质量。

## 二、跨板块溢出分析架构

### 2.1 设计背景

当前 16 个板块独立分析，没有板块间传导机制。一条能源新闻对航空的影响完全靠 LLM 通识知识，没有历史传导系数支撑。

### 2.2 SPILLOVER_MAP 先验（engine/news.py）

16个板块的静态传导关系表，格式：`{目标板块: [(来源板块, 代表性ticker, 传导描述)]}`，共约30条传导关系，覆盖主要金融传导路径：

- 利率 → 成长股估值 / REITs 折现率
- 能源价格 → 工业利润率 / 消费成本
- 中国经济 → 新加坡贸易 / 半导体供需
- 收益率曲线 → 银行净息差

### 2.3 build_spillover_context()（engine/news.py）

对目标板块查询传导来源，抓取每个来源板块的最新标题，拼装成结构化溢出信号块注入 `news_ctx`。

输出示例：
```
【跨板块溢出信号 · 传导至 科技成长(纳指)】
  ↳ 美国长债  [无风险利率上行 → 成长股折现率压缩估值]  [先验]
     • Treasury yields surge to 4.8%  —  Bloomberg
  ↳ AI算力/半导体  [算力供给景气 → 科技板块业绩预期]  [先验]
     • NVIDIA data center demand accelerates  —  Reuters
```

注入点：Tab3 `_run_sector_analysis`、Tab4 Quant Audit，追加在本板块新闻之后，原有纵向流程不动。

### 2.4 SpilloverWeight 学习层（engine/memory.py）

**设计原则**：手动先验负责"哪些板块之间存在传导关系"（结构性，变化慢），学习层负责"传导强度有多大"（随环境动态调整）。数据不足时静默，达到阈值自动激活，无缝降级。

**SpilloverWeight 表字段**：
`source_sector`, `target_sector`, `macro_regime`, `correlation`（Pearson r）, `conflicts_prior`（是否与先验方向相反）, `sample_count`

**update_spillover_weights()**：
- 对同一 `macro_regime` 内所有板块对计算 `accuracy_score` 的 Pearson 相关系数
- 激活条件：每对组合同期已验证决策 ≥ 15 条
- 互斥判定：已知先验对且 r < −0.30 → `conflicts_prior=True`
- 每次 `run_meta_agent_analysis` 运行时自动触发

**get_spillover_weights()**：
- 只返回 `sample_count >= 15` 的已激活记录
- 数据不足返回空列表，调用方降级到先验

**build_spillover_context() 两层逻辑**：
1. 优先调用学习层，按 |r| 排序，最多取4个来源
2. 数据不足时降级到 SPILLOVER_MAP 先验
3. 互斥时追加：`⚠ 方向与先验相反，传导关系存疑` → 红队自行判断

## 三、Bug 修复与数据安全

### 3.1 数据泄露修复（engine/history.py）

`run_sector_backtest` 中 `get_historical_context` 调用缺少 `cutoff_date` 参数，导致早期回测点可以看到未来已验证决策，回测准确率虚高。

```python
# 修复前
hist_ctx = get_historical_context("sector", sector_name=sector, macro_regime=..., n=3)

# 修复后
hist_ctx = get_historical_context("sector", sector_name=sector, macro_regime=..., n=3, cutoff_date=date)
```

注：`run_walk_forward_backtest` 原本已正确传入 `cutoff_date`，本次仅修复普通回测。

### 3.2 其他 Bug 修复

| # | 问题 | 严重性 | 修复 |
|---|------|--------|------|
| 1 | `learning_log.dormant` 列和 `SpilloverWeight` 表在旧数据库上不存在 | 致命 | `_migrate_db` 补充迁移逻辑 |
| 2 | 学习层 ticker fallback 错误使用 `SECTOR_QUERY_MAP.keys()[0]` | 致命 | 改为 `src`（用板块名查询） |
| 3 | 列表推导式变量 `d` 遮蔽外层 `DecisionLog` 循环变量 | 中 | 改为 `dt` |
| 4 | 旧 `LearningLog` 行 `dormant=NULL` 被 `== False` 过滤掉 | 致命 | 加 `OR NULL` 条件 |
| 5 | `human_label != "black_swan"` 对 NULL 行不安全 | 中 | 加 `OR IS NULL` 条件（两处） |

### 3.3 数据泄露分析结论

经系统性排查，以下问题经分析不构成实际泄露：

**批次不独立问题**：`run_meta_agent_analysis` 提取 pattern 是离线全局统计过程，不是模型训练。`get_historical_context` 在走势回测中传入 `cutoff_date`，`get_learning_patterns` 有 `created_at < cutoff_date` 过滤，meta-agent 在回测开始前产生的 pattern `created_at` 为今天，早于任何历史 `cutoff_date`，会被正确过滤。`cutoff_date` 机制已覆盖此场景。

## 四、完整运行路径

```
启动
  └─ init_db() → create_all 建新表 + _migrate_db 补旧表列

板块分析（Tab3/Tab4）
  └─ build_context()         本板块新闻
  └─ build_spillover_context()
       ├─ get_spillover_weights()  数据充足 → 学习系数，互斥时附冲突标记
       └─ SPILLOVER_MAP 先验       数据不足 → 静默降级
  └─ news_ctx 合并注入红队 prompt

Meta-Agent（Admin）
  └─ 验证决策 ≥ 25 才执行
  └─ CI 过滤 + 证据反驳已有 pattern
  └─ Regime 休眠检测
  └─ 保存新 pattern
  └─ update_spillover_weights()

学习层激活后（每对板块 ≥ 15 条同期决策）
  └─ build_spillover_context 切换到学习系数
  └─ 互斥时追加冲突标记，红队自行判断
```

## 七、数据断点与逻辑链条修复

### 7.1 移除 `quant_results["returns"]`（engine/agent.py）

research_node 每次运行都将完整 pandas DataFrame（历史价格序列）存入 `quant_results["returns"]`，随后随 `audit_cache` 进入 session_state。该字段从未被任何下游节点或 UI 代码读取，纯属内存浪费。已移除，改为注释说明。

原则：重要的结构化数据存数据库，大型中间计算结果不进 session_state。

### 7.2 修正 AgentState TypedDict（engine/agent.py）

`final_memo` 字段定义了但从未被赋值或读取（死字段）。实际使用的是 translator 节点产出的 `audit_memo`，但该字段未在 TypedDict 中声明，导致类型定义与运行时行为不一致。已将 `final_memo` 替换为 `audit_memo`。

### 7.3 修复反思推理链 DB 恢复断点（engine/memory.py）

`get_all_today_audit_records` 负责页面刷新后从数据库重建 `audit_cache`，但未读取 `reflection_chain` 字段，导致用户刷新页面后反思推理链丢失。已补充 `r.reflection_chain` 的读取与映射，DB 恢复路径现在与首次运行路径完全对齐。

## 八、Backtest 系统完善

### 8.1 Quota 耗尽断点续训（engine/history.py · engine/memory.py · pages/admin.py）

免费 API（Gemini Free Tier）在回测过程中可能中途耗尽额度。原来遇到 quota 错误直接中断，已完成进度全部丢失。

**新机制**：
- `DecisionLog` 新增 `needs_retry` 布尔字段（默认 `False`）
- `save_decision()` 接收 `needs_retry` 参数并写入 DB
- quota 耗尽时：保存当前 `(sector, date)` 的存根记录（`needs_retry=True`，无 `ai_conclusion`），设 `_quota_hit=True`，中断循环
- `run_sector_backtest` 返回值改为 `(results, _quota_hit)` 元组
- 续训时：`_done_pairs` 查询过滤 `needs_retry=False`，存根不被误判为已完成，该 pair 会被重新处理
- Admin 显示 quota 耗尽提示，CSS 样式统一使用 `var(--text)` / `var(--muted)` / `var(--accent)` CSS 变量，与项目主题一致

### 8.2 进度条续训闪烁修复（engine/history.py）

续训时每个已跳过的 pair 均触发一次 `progress_cb`，Streamlit 合并快速更新导致进度条视觉上一闪而过。

**修复**：改为在循环开始前统计所有已完成 pair 数，一次性上报初始进度偏移，后续只对实际处理的 pair 调用 `progress_cb`。效果：进度条从断点位置开始平滑推进，不再闪烁。

### 8.3 回测日期范围修正（pages/admin.py）

`date_input` 未设置 `min_value` 时默认只允许从 `value` 往前翻 10 年（Streamlit 行为），导致最早只能选到 2010 年。

经数据源排查：
- FRED 宏观数据：1954 年起
- yfinance 价格/VIX：多数 ETF 1998–2000 年起
- **GDELT V2 新闻 API：2015 年 2 月起**（约束边界）

新闻上下文是回测质量的核心，2015 年之前无数据意味着训练质量严重下降。所有四个日期选择器统一设 `min_value=2015-03-01`，与 GDELT 覆盖范围匹配。

### 8.4 Backtest Review 面板（pages/admin.py · engine/memory.py）

Admin 新增 **Backtest Review** 独立 Tab，持久化展示所有已完成回测记录（刷新不丢失）。Simple Backtest 与 Walk-Forward 记录统一展示，Phase 列区分。

**面板结构（从上到下）**：
1. 三个筛选器：板块 / 宏观周期 / Phase（`simple` / `train` / `test`）
2. 记录汇总表格：日期、板块、周期、方向、VIX、置信度、Phase、持仓期
3. 危险操作 · 删除面板（折叠 expander，双重保护）
4. 完整分析记录：下拉选择任意记录，expander 内渲染完整 AI 分析（`st.markdown` 正确渲染 Markdown 格式）

**新增函数**：
- `get_backtest_records(sector, regime, phase, limit=500)`：从 DB 读取，排除 `needs_retry` 存根
- `delete_backtest_records(sector, phase)`：按范围删除，返回删除行数

**AI 分析全文渲染修复**：原用 `st.text_area`，`***` 等 Markdown 符号以原始文本显示；改为 `st.markdown` 后正确渲染。

**置信度为空修复**：Simple Backtest 的 prompt 缺少 `[XAI_ATTRIBUTION]` 指令块，LLM 不输出置信度字段。已补全该指令块，与 Walk-Forward prompt 对齐，新跑记录将正常填充置信度。

### 8.5 回测记录删除功能（双重保护）

删除操作需同时满足两个条件才激活"确认删除"按钮：
1. 勾选确认 checkbox（"我已了解此操作不可撤销"）
2. 输入管理员密码（从 `secrets.toml` 的 `LOGIN_PWD` 读取，不暴露在源代码中）

支持按板块 + Phase 精准删除，或全量清空。删除后 `st.rerun()` 自动刷新列表。

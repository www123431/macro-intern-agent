## 二、统计量化升级：非对称置信区间

### 2.1 设计依据

传统金融风险指标（Sharpe、VaR）基于正态分布假设，输出对称置信区间。但金融数据的真实特征是：
- 股票收益率轻微**左偏**（崩盘比上涨更快）
- Sharpe Ratio 的抽样分布在 SR > 0 时**右偏**
- VaR 的估计偏差方向应保守（宁可高估风险）

正确做法是使用非对称置信区间，而非强加正态假设。

### 2.2 Bootstrap 非参数 CI（`engine/quant.py`）

新增 `_bootstrap_ci(series, stat_fn, n_boot=1000, alpha=0.05)`：
- 对收益率序列进行 1000 次有放回重采样
- 每次计算目标统计量（VaR 或 Sharpe）
- 取第 2.5 和 97.5 百分位作为上下界
- **无需假设任何分布**，区间天然非对称

### 2.3 Cornish-Fisher 调整 VaR（`engine/quant.py`）

新增 `QuantEngine.cornish_fisher_var(returns, alpha=0.05)`：

```
z_cf = z + (z²-1)·S/6 + (z³-3z)·K/24 − (2z³-5z)·S²/36
VaR  = μ + z_cf × σ
```

- S = 偏度，K = 超额峰度，z = 标准正态分位数（-1.6449）
- 左偏（崩盘倾向）的资产 → VaR 比正态假设**更大（更保守）**
- `use_cf=False` 默认关闭，等数据量 > 200 条后手动启用

### 2.4 `AnalyticsEngine.calculate_metrics()` 扩展

新增返回字段：

| 字段 | 含义 |
|---|---|
| `var_ci` | Bootstrap 95% CI for VaR（左尾更宽） |
| `sharpe_ci` | Bootstrap 95% CI for Sharpe（SR>0 时右偏） |
| `var_cf` | Cornish-Fisher 调整 VaR（参考值，始终计算） |
| `skewness` | 样本偏度 |
| `excess_kurt` | 样本超额峰度 |

### 2.5 UI 展示（`ui/tabs.py`）

Tab4 Quant Audit 结果区域：
- Row 2 从 2 列扩展为 3 列，新增 **Sharpe Ratio** 独立卡片

  - Bootstrap 95% 非对称 CI
  - 偏度/超额峰度数值

## 三、量化自学习闭环：三层架构

### 设计原则

> 系统不仅要学习新闻→判断→结果的规律，也要学习**量化状态→判断→结果**的规律。这样 AI 的分析不再纯粹依赖新闻叙事，而是有客观量价数据和历史条件胜率作为校验维度。

### Layer 1：量化上下文注入（`engine/quant.py`）

**`build_quant_context(sector_etf, as_of_date)`**

每次回测分析前，生成结构化量化指标块注入 prompt，所有数据严格使用 `as_of_date - 1` 之前的数据（零泄露）：

```
[QUANT_CONTEXT]
板块ETF: XLF
价格动量：1M +3.2% | 3M -1.8% | 6M +12.4%
相对强弱 vs SPY（1M）：+1.6%（跑赢大盘）
RSI(14)：67 — 中性区间
布林带位置：72% — 中性区间
信用利差(HYG-IEF 20日价差)：+0.8% — 收窄→风险偏好改善
[/QUANT_CONTEXT]
```

**`compute_state_vector(sector_etf, as_of_date)`**

将价格状态离散化为三个维度，用于 Layer 2 状态变化检测：

```python
{
    "momentum_regime": "上升" | "震荡" | "下降",   # 1M 回报 ±2% 阈值
    "rsi_zone":        "超买" | "中性" | "超卖",    # RSI 70/30 阈值
    "vol_regime":      "高波动" | "正常" | "低波动", # 20日 vs 历史波动率
}
```

### Layer 2：状态变化驱动采样（`engine/history.py`）

**解决的问题**：按日历频率回测时，牛市中同一板块可能连续生成 30 次"超配"信号，造成标签不平衡和时序自相关，本质上是把同一个样本喂了 30 遍。

**机制**：
- 维护每个板块的 `_last_state` 和 `_last_signal_date`
- 只有满足以下任一条件时才生成新信号：
  - 状态向量任意维度发生变化 AND 距上次 ≥ 7 天（最小间隔防抖）
  - 距上次 ≥ 90 天（季度兜底，防止完全沉默）
- 跳过的配对显示 "↷ 跳过·状态未变"，不消耗 API quota

**效果**：同一趋势内只在起始、超买、回调等关键节点各生成一次信号，天然解决标签不平衡问题。

**Prompt 新增分析框架**：
```
### 3. 量价背离检验
[对比新闻情绪与量化指标（动量、RSI、信用利差），识别是否存在叙事与价格走势的背离]
```

### Layer 3：Purged Walk-Forward（`engine/history.py`）

**当前状态**：代码已实现，休眠中。

**激活条件**：`get_verified_decision_count() >= 300`

**激活效果**：
- train/test 边界加 5 天 embargo 期（防止边界附近的信息泄漏）
- 每条记录的 `debate_transcript` 存入 `purged_wf_active` 和 `verified_count_at_run` 便于回溯

**量化背景**：Lopez de Prado《Advances in Financial Machine Learning》中的标准时序交叉验证方法，专门解决金融数据自相关导致的 CV 失效问题。

### QuantPatternLog：条件胜率表（`engine/memory.py`）

**本质**：一张 `状态向量 × 宏观周期 × 方向 → 历史胜率` 的频率统计表，是量化实务中的条件信号评估（Conditional Signal Evaluation）。

**数据库模型**：

```
QuantPatternLog
├── state_fingerprint  : "动量:下降|RSI:超买|波动:高"
├── macro_regime       : "高波动收缩"
├── direction          : "超配"
├── total_count        : 13
├── correct_count      : 4
├── partial_count      : 3
├── accuracy_rate      : 0.31
├── avg_accuracy       : 0.42
└── last_updated       : 2026-04-10
```

**写入时机**：`verify_pending_decisions()` 完成每条记录的价格验证后，调用 `_update_quant_pattern(dec, session)` 滚动更新。

**读取时机**：每次回测/实时分析时，调用 `get_quant_pattern_context(state_vector, macro_regime, cutoff_date)` 生成注入文本：

```
[QUANT_PATTERN_PRIOR]
历史状态组合：动量:下降|RSI:超买|波动:高 × 高波动收缩
样本总量：13 条已验证决策
各方向历史准确率：超配 31%（n=13, 平均得分=0.42）
⚠ 注意：该状态下超配判断历史胜率偏低（31%），建议提高风险敞口评估权重、降低置信度。
[/QUANT_PATTERN_PRIOR]
```

**数据泄露防护（三道防线）**：

| 防线 | 机制 |
|---|---|
| 量化指标计算 | `as_of_date - 1` 截止，零泄露 |
| 历史决策上下文 | `cutoff_date` 过滤，与现有机制一致 |
| 量化模式先验 | `last_updated <= cutoff_date` 查询约束 |
| 验证本身 | 价格数据 T+N 后才抓取，天然时间缓冲 |

**激活预期**：

| 样本量 | 期望效果 |
|---|---|
| < 50 | 大多数状态组合 n < 3，不输出先验 |
| 50–150 | 少数高频组合（震荡+中性+正常）开始有参考值 |
| 150–300 | 多数组合稳定，模式开始可信 |
| > 300 | 可分宏观周期分层，进入 Purged WF 模式 |

## 五、语义漂移监控（今日新增）

### 5.1 背景

Meta-Agent（LSE 指令合成）随时间迭代时，可能出现：
- 同一板块 × 宏观周期组合，先后生成相互矛盾的指令（力度强弱均有）
- 早期低质量规则变为"休眠"后仍在 UI 可见，未被及时归档
- 无人工监控时，矛盾规则可能同时影响后续分析

### 5.2 实现（Admin Learning Tab 新增区域）

新增 **Semantic Drift Monitor** 区块：

- **三维过滤器**：按板块 / 宏观状态 / 类型筛选 LearningLog 条目
- **矛盾检测器**：自动识别同一 `sector × regime` 下同时存在"强弱"两类 bias 规则的情况，以红色 `⚠ 矛盾` 标记
- **状态标签系统**：
  - `[已归档]`：applied=True
  - `[休眠]`：dormant=True
  - `[⚠ 矛盾]`：与同类条目存在方向冲突
- **时序排列**：所有条目按创建时间升序展示，便于追踪规则演变轨迹

### 5.3 数据支持

`get_learning_log_raw(limit=200) → list[dict]`（`engine/memory.py` 新增）：返回全量 LearningLog 条目，包含 `created_at`、`sector`、`regime`、`type`、`description`、`samples`、`applied`、`dormant` 字段，供监控界面消费。

## 八、待解决的设计问题：压力情景经验如何回流学习闭环

### 8.1 核心矛盾

当前 Stress Test 的分析结果仅存于 session，不写入任何数据库。这带来一个设计缺口：

> 黑天鹅测试的价值不只是"模拟一下看看"，而是让系统在极端情景真实发生之前积累"见过"的经验，从而在真实事件中判断更有依据。

但直接写入现有的学习表存在**数据污染风险**：

- `QuantPatternLog` 的胜率依赖"AI 预测 + N 天后真实价格验证"的闭环
- 压力情景没有对应的真实价格结果，准确率永远无法被验证
- 混入真实样本会稀释信噪比，破坏胜率表的可信度

### 8.2 本质问题

**压力情景没有 ground truth。** 真实决策可以用价格结果打分，但假设情景的"对不对"取决于那个情景是否真实发生——而黑天鹅的定义恰恰是低频、难以预测。

### 8.3 三个可能的解法方向（待讨论，暂不实现）

| 方向 | 机制 | 优点 | 缺点 |
|---|---|---|---|
| **事后对照** | 真实市场出现近似情景时，手动将沙盒推演与真实结果对比打分，再决定是否写入 | 有真实验证，不污染数据 | 依赖人工干预，时效性差 |
| **隔离存储** | 新建 `StressPatternLog`，与 `QuantPatternLog` 分开；AI 分析时两张表分别注入，权重不同 | 不污染真实样本，沙盒经验可被参考 | 需要设计合理的权重衰减机制 |
| **只学结构，不学结论** | 不写入方向性预测，只将"极端情景下应关注的传导路径"写成 LearningLog 条目 | 最安全，框架级别的知识不依赖验证 | 丢失了具体方向判断的积累 |

### 8.4 当前决策

**暂不实现**，原因：
1. 数据量积累才是当前主要瓶颈，真实验证样本尚不足以支撑对极端情景做有意义的分类
2. 上述三个方向都需要进一步讨论 ground truth 的定义
3. 待真实验证样本达到 150–200 条后，极端状态下的真实样本会增多，届时再评估哪个方向成本收益最优

## 九、完整量化学习数据流（截至今日）

```
回测日期 T
  │
  ├─ compute_state_vector(etf, T)        → T-1 日价格（零泄露）
  ├─ build_quant_context(etf, T)         → T-1 日价格 + 技术指标
  ├─ get_historical_context(cutoff=T)    → 只看 T 前已验证决策
  ├─ get_quant_pattern_context(          → 只查 last_updated ≤ T 的行
  │    state, regime, cutoff=T)
  │
  └─ 生成分析 → save_decision(
                  debate_transcript = {state_vector, purged_wf_active, ...}
                )

T + 验证窗口后（5/40/90 天）
  │
  └─ verify_pending_decisions()
       ├─ accuracy_score 计算
       ├─ _feedback_routing_weights()    → 更新 NewsRoutingWeight
       └─ _update_quant_pattern()        → 更新 QuantPatternLog

Meta-Agent 运行时
  └─ run_meta_agent_analysis()
       ├─ 证据反驳 + Regime 休眠检测
       └─ update_spillover_weights()
```

## 十、多源数据时间戳对齐

### 10.1 问题背景

项目迭代后，AI 分析同时使用三类数据：

| 类型 | 来源 | 性质 |
|---|---|---|
| 主观新闻 | GDELT / Alpha Vantage / RSS | 非结构化，有发布时间戳 |
| 客观量化指标 | yfinance 价格数据 | 结构化，以交易日为单位 |
| 宏观经济数据 | FRED | 结构化，有发布滞后（publication lag） |

如果三类数据的截止时间不一致，会发生**交叉数据泄漏**：用已经被价格消化过的新闻预测已经发生的结果，或用尚未发布的宏观数据影响回测判断。

### 10.2 现有机制（已正确处理）

| 数据流 | 截止机制 | 实现位置 |
|---|---|---|
| 价格/量化指标 | `price_cutoff = date - 1` | `build_snapshot` |
| GDELT 新闻 | T-1 截止，内部强制 | `get_gdelt_news` |
| FRED 宏观数据 | 每个指标独立 publication lag（CPI=35天、失业率=7天、利率=2天） | `_FRED_PUB_LAG` + `_get_fred_value_on` |
| `build_quant_context` | `cutoff = as_of_date - 1` | `engine/quant.py` |

### 10.3 FRED vintage 问题（已知，暂接受）

FRED 数据存在 **数据修订版本（vintage）** 问题：
- 回测时调用 FRED API 返回的是**当前最新修订值**
- 历史上该日期能看到的是**当时的初稿值**，可能相差 0.3–0.5 个百分点
- 这在宏观拐点时可能影响 regime 判断

当前决策：接受这个误差。解决方案需要 FRED vintage API（`realtime_start` 参数），成本较高，等数据量积累后再评估必要性。

### 10.4 今日新增的修复（`engine/history.py`）

**1. `data_cutoff_log` 显式记录 `quant_context_cutoff`**

原来审计日志只记录 `price_cutoff` 和 `gdelt_cutoff`，量化指标的截止日期虽然逻辑上等于 `price_cutoff`，但没有显式标注。现在补充：
```python
"quant_context_cutoff": str(price_cutoff),  # build_quant_context 使用同一 T-1 截止
```

**2. 跨数据流对齐检查**

在 `build_snapshot` 中新增对齐检查：若 `gdelt_cutoff` 与 `price_cutoff` 偏差超过 1 天（可能发生于周末或节假日边界），记录 `logger.warning` 并在 `data_cutoff_log` 写入 `alignment_warning` 字段，便于事后审计。

**3. 简单回测 `debate_transcript` 补全**

原来简单回测的 `debate_transcript` 只存 `data_cutoff_log`，缺少 `state_vector` 和 `verified_count_at_run`，与 Walk-Forward 版本不一致。现已补全，两种回测模式的审计信息结构一致：
```python
debate_transcript={
    "data_cutoff_log":       snap.get("data_cutoff_log"),
    "state_vector":          _curr_state,
    "purged_wf_active":      False,
    "verified_count_at_run": get_verified_decision_count(),
}
```

### 10.5 关于新闻数据源扩充

当前三层新闻来源（GDELT 回测主力 + Alpha Vantage 实时 + RSS 兜底）已满足基本需求。扩充数据源的优先级排序：

| 优先级 | 来源 | 理由 |
|---|---|---|
| ★★★ | FRED 更多指标（PMI、零售销售） | 免费，时间戳精确，有 publication lag 可控 |
| ★★★ | SEC EDGAR 8-K 重大事项披露 | 事件驱动，时间戳精确，免费 |
| ★★ | Alpha Vantage 扩大用量（覆盖回测） | 已有情感打分，对齐机制已有 |
| ★ | Bloomberg / Reuters API | 质量最高，但成本过高，不适合当前阶段 |

**当前结论**：先保证现有三类数据的时间戳对齐质量，再考虑数据源扩充。数量不是瓶颈，信号密度和对齐准确性才是。

## 十一、训练进度 Tab：季度 / 月度覆盖率总览

### 11.1 背景

回测去重逻辑保证了每个 `(tab_type, 板块, 日期)` 三元组只会被写入一次。这意味着训练本质上是有终点的——从 2015-03-01 到当天，理论上所有板块 × 所有日期点都能被穷举覆盖。因此可以给用户展示一个全局训练进度，告知距离"完全覆盖"还差多少。

由于季度训练集（QS）与月度训练集（MS）是完全独立的去重命名空间（`tab_type` 不同），二者的覆盖率需要分开统计，合并展示会导致数字混乱。

### 11.2 变更内容

**`pages/admin.py`**：

1. **新增 Tab** — 在 "Backtest Review" 与 "Stress Test" 之间插入第六个 Tab `"训练进度"`，对应变量 `_tab_coverage`。

2. **移除 Backtest Tab 内的覆盖率面板** — 原先的覆盖率区块仅展示当前回测所选频率和板块的数据，耦合度高、位置错误，已从 `_tab_bt` 中完整删除。

3. **新增 `_render_coverage_section(freq, label_cn, accent)` 辅助函数** — 封装单个频率的完整渲染逻辑：
   - 调用 `get_training_coverage(sectors=ALL_SECTORS, freq=freq, start_date="2015-03-01", end_date=today)`
   - 渲染整体进度条（颜色随完成率变化：绿 ≥ 80% / 橙 ≥ 40% / 红 < 40%）
   - 渲染按完成率升序排列的板块卡片网格（3 列，标注"完成 / 进行中 / 未开始"状态）
   - 渲染已覆盖宏观制度分布徽章

4. **双列布局** — `_tab_coverage` 使用 `st.columns(2, gap="large")`：
   - 左列：`_render_coverage_section("QS", "季度", "#6366f1")`（靛蓝色调）
   - 右列：`_render_coverage_section("MS", "月度", "#0ea5e9")`（天蓝色调）
   - 底部 caption 标注数据范围、板块总数、计算口径

5. **全板块视角** — 覆盖率统计始终使用完整的 `SECTOR_ETF` 板块列表，与当前回测的选板块无关，反映系统整体训练进度。

### 11.3 覆盖率计算口径说明

| 维度 | 说明 |
|---|---|
| 起始日期 | 2015-03-01（GDELT 覆盖最早边界） |
| 结束日期 | 当天（`datetime.date.today()`，动态）|
| 计数单位 | 唯一 `(板块, 采样日期)` 对，以季度或月频采样 |
| 去重条件 | `tab_type` 匹配 + `needs_retry=False` + `ai_conclusion IS NOT NULL` |
| QS / MS 独立 | 两个频率互不干扰，各自有独立的理论总量和已完成数 |

## 十二、Gemini 号池覆盖全部 AI 调用路径

### 12.1 背景

`engine/key_pool.py` 中已实现完整的 Gemini 号池（`KeyPoolManager`），支持 quota 错误自动轮转、空输出熔断、每日重置。但以下两处 AI 调用绕过了号池，直接使用单 Key：

| 位置 | 问题 |
|---|---|
| `engine/agent.py` 4 个节点 | 通过闭包直接使用传入的 `model` 对象，无轮转能力 |
| `app.py` 启动初始化 | 手动读取 `GEMINI_KEY` 单 Key，绕过号池单例 |

### 12.2 变更内容

**`engine/agent.py`**：

新增 `_pool_call(prompt: str) -> str` 封装函数（定义在 `build_agent_graph` 内部，作为闭包），统一所有节点的 AI 调用：

```
_pool_call 调用链：
  get_pool()                   → 获取号池单例（与回测侧共用）
  _pool.get_model()            → 用当前活跃 Key 构造 GenerativeModel
  model.generate_content(...)  → 实际 API 调用
  _pool.report_success()       → 成功后重置连续失败计数

  发生 quota 错误时：
    _pool.report_quota_error()  → 内部递增失败计数，达到阈值则轮转 Key
    _pool.get_model()           → 切换到新 Key 后重试一次

  所有 Key 耗尽 (AllKeysExhausted)：
    返回中文提示字符串（不崩溃），引导用户前往 Key Pool Manager

  空输出熔断 (EmptyOutputCircuitBreaker)：
    返回熔断提示字符串

  其他异常：
    返回 "⚠️ AI 分析引擎暂时离线" + 错误摘要
```

原来 4 个节点各自独立的 `try/except model.generate_content(...)` 全部替换为 `_pool_call(prompt)`：
- `red_team_node` — 量化风险审计（主分析节点）
- `reflection_node` — 红队拦截后的替代资产反思
- `technical_audit_node` — 硬核技术报告
- `translator_node` — CEO 级分层简报

**`app.py`**：

```python
# 旧：绕过号池，单独读 GEMINI_KEY
GEMINI_KEY = st.secrets.get("GEMINI_API_KEY") or st.secrets.get("GEMINI_KEY")
genai.configure(api_key=GEMINI_KEY)
model = genai.GenerativeModel(model_name="gemini-2.5-flash")

# 新：走号池单例，与 admin.py 回测侧共享同一个池
_pool = get_pool()
model = _pool.get_model()
```

两种失败情况分别给出独立提示并 `st.stop()`：
- `AllKeysExhausted`：提示所有 Key 已耗尽，引导添加新 Key 或等明天重置
- 其他异常：提示配置文件读取失败，引导检查 `secrets.toml`

同时移除了已无用的 `import google.generativeai as genai`（`get_model()` 内部已处理 configure）。

### 12.3 架构一致性

修改后，整个系统所有 Gemini 调用路径统一走同一个号池单例：

```
app.py 启动               → get_pool().get_model()
agent.py 实时分析节点      → _pool_call() → get_pool()
admin.py 回测 (backtest)  → get_pool()  (既有，不变)
admin.py 压力测试          → get_pool()  (既有，不变)
```

Key 池状态（已调用次数、连续失败、exhausted/active 状态）在整个 Streamlit 进程内共享，不会因页面切换而重置。号池 stats 持久化到 `.streamlit/key_pool_stats.json`，重启后恢复。

### 12.4 注意事项

新闻 API（Alpha Vantage、GNews）保持原有单 Key 模式不变，这两个接口有各自的固定 Key，暂不需要轮转机制。

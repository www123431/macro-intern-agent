# 开发日志 · 2026-04-11

## 概览

本次开发集中在两个方向：**统计鲁棒性与过拟合防御**、**训练/测试集硬性划分**。核心动机是在现有工程框架已较为严谨的基础上，解决方法论层面的根本性问题——即系统的学习闭环是否在用已见过的数据来自我验证。

---

## 一、背景：量化分析视角的系统审查

在本次开发前，对系统进行了完整的量化方法论审查，识别出以下核心问题：

| 问题类型 | 具体表现 |
|---|---|
| 信号时效性 | 公开新闻在发布后数秒内被市场定价，T-1 新闻信号存在结构性时效劣势 |
| LLM 隐性知识污染 | 模型训练数据包含历史市场结果的叙事，回答中混入事后知识 |
| 验证窗口任意性 | 5/40/90 天三个固定窗口，同一信号用不同窗口结论可能相反 |
| 小样本过度自信 | 冷门板块桶样本 n < 5，QuantPatternLog 胜率数字无统计意义 |
| 测试集污染 | LearningLog / QuantPatternLog / SkillLibrary 的滚动更新，使整段历史数据都变成了隐性训练集 |

针对上述问题，讨论了五个统计鲁棒性工具（对数收益率切换、镜像反事实审计、三重障碍法、James-Stein 收缩估计、置换检验 + Bonferroni 校正）的可行性，并得出结论：**所有这些工具的有效性都以"测试集干净"为前提**，因此训练/测试集划分是优先级最高的基础性工作。

## 二、训练/测试集硬性划分

### 2.1 设计原则

```
数据时间轴：
─────────────────────────────────────────────────────────────
2015-03-01                    2023-01-01              今日
│◄────────── 训练集 ──────────►│◄──── 测试集（封存）────►│

规则：
• 训练集：回测、验证、学习写回均允许
• 测试集：回测可运行，验证可打分，但所有学习写回全部拦截
• TRAIN_TEST_CUTOFF 一旦设定，永远不得向前移动
```

**分界日期：`2023-01-01`**

选择 2023-01-01 的理由：
- 训练集覆盖 2015-2022 年（8 年），包含完整的低利率→加息周期、两次大跌（2018Q4、2020）、加息周期起点
- 测试集覆盖 2023 年至今（3 年+），包含高利率稳定期、AI 板块重估、地区银行危机等与训练集结构不同的制度

### 2.2 变更内容

**`engine/memory.py`**：

新增常量：

```python
TRAIN_TEST_CUTOFF: datetime.date = datetime.date(2023, 1, 1)
```

在以下 5 个"学习写回"函数中增加测试集门控，凡是 `decision_date >= TRAIN_TEST_CUTOFF` 的记录，验证打分正常进行，但写回操作直接 return：

| 函数 | 写入目标 | 门控位置 |
|---|---|---|
| `_update_quant_pattern()` | `QuantPatternLog` | 函数入口，`decision_date` 检查 |
| `_feedback_routing_weights()` | `NewsRoutingWeight` | 函数入口，`decision_date` 检查 |
| `_check_horizon_mismatch()` | `LearningLog` | 方向检查之后，早期 return |
| `run_meta_agent_analysis()` | `LearningLog` | DB 查询层加 `< TRAIN_TEST_CUTOFF` 过滤 |
| `maybe_update_skill()` | `SkillLibrary` | DB 查询层加 `< TRAIN_TEST_CUTOFF` 过滤 |

`decision_date` 的取值逻辑（所有门控统一）：
```python
_dec_date = dec.decision_date or (dec.created_at.date() if dec.created_at else None)
if _dec_date is not None and _dec_date >= TRAIN_TEST_CUTOFF:
    return  # 测试集，拦截写回
```

| 操作 | 是否受限 | 说明 |
|---|---|---|
| 在测试集日期运行回测 | ❌ 不受限 | 需要生成结论才能评估 |
| 测试集记录的价格验证打分 | ❌ 不受限 | 需要 `accuracy_score` 才能度量性能 |
| `get_historical_context()` 读取 | ❌ 不受限 | 已有 `cutoff_date` 机制控制时序隔离 |
| `get_quant_pattern_context()` 读取 | ❌ 不受限 | 已有 `cutoff_date` 过滤 |
| 所有 5 个学习写回函数 | ✅ 拦截 | 测试集结果不得回流学习表 |

**`pages/admin.py`**：

在 Simple Backtest 的日期选择器下方，新增训练/测试集状态指示条，根据所选日期范围动态显示三种状态：

| 状态 | 颜色 | 触发条件 |
|---|---|---|
| ✅ 完全在训练集内 | 绿色 | `bt_end < TRAIN_TEST_CUTOFF` |
| 🔒 完全在测试集内 | 靛蓝 | `bt_start >= TRAIN_TEST_CUTOFF` |
| ⚠️ 跨越边界 | 橙色 | `bt_start < TRAIN_TEST_CUTOFF <= bt_end` |

跨越边界时，指示条明确说明哪段写入学习表、哪段仅打分，防止用户误以为整段回测都会触发学习。

## 三、统计工具实装状态总览

以下五个工具经过讨论，确认方向正确，优先级排序如下：

### 优先级 HIGH（直接改善方法论）

**三重障碍法 (Triple-Barrier Method)** ✅ 已实装（§六）
- 用止盈/止损/时间三条边界替代固定窗口验证
- 解决验证窗口任意性问题
- 参数需用经济逻辑锚定，建议：止盈 = 1σ，止损 = 0.7σ，时间障碍 = 2× 信号半衰期

**置换检验 + 时序块自举 (Block Bootstrap Permutation)** ✅ 已实装（§九）
- 10000 次块级随机重排建立噪声基准，检验信号是否显著优于随机
- 必须用移块自举（块长 3-6 个月）而非 i.i.d. 置换，保留时序结构
- Bonferroni 校正：16 板块并发时单板块 p 值阈值从 0.05 压缩至 0.003
- 采用 Romano-Wolf stepdown（比 Bonferroni 更适合有相关性的多检验场景）
- 门控阈值 `_MIN_N_PERMUTATION = 100`：样本不足时返回 `insufficient_data`，不产生误判

### 优先级 MEDIUM（改善估计质量）

**James-Stein 收缩估计 (Empirical Bayes Shrinkage)**
- 样本 n < 阈值的板块桶强制向全局先验收缩
- 前提：置换检验先确认有真实 alpha 后，再用此工具防止小样本过度自信

**对数收益率切换**
- 将全量指标计算从简单收益率切换为对数收益率
- 影响面广（`engine/quant.py` 多处），需统一改造，不宜零散修改

### 优先级 LOW（辅助质量控制）

**镜像反事实审计 (Mirror Auditing + LCS)**
- 对每次 AI 输出构造镜像场景，检验逻辑一致性分数（LCS ≥ 0.7）
- 注意：只能过滤"逻辑退化"（无论输入什么都给同一结论），不能证明通过测试的信号有真实 alpha

## 五、LCS 全套方案实装

### 5.1 架构概述

在已建立的训练/测试集划分基础上，本次引入两条独立的质量控制链：

```
LCS 质量门（behavioral consistency gate）
  ← 阻止逻辑退化的分析写入 LearningLog / QuantPatternLog

Clean Zone 前瞻隔离边界（foreknowledge isolation gate）
  ← 区分"受 LLM 历史记忆污染"与"真实 out-of-sample"的数据

两条链独立运作，互不替代：
  LCS 通过 ≠ 无污染（可能记忆了历史走势但仍能逻辑自洽地回答镜像题）
  Clean Zone 日期 ≠ 一定高 LCS（后训练数据也可能产生退化逻辑）
```

### 5.2 新文件：`engine/lcs.py`

包含四个核心组件，组成双门控架构：

| 组件 | 函数 | 作用 |
|---|---|---|
| 镜像测试 | `run_mirror_test()` | 翻转牛熊输入，检验结论是否随之翻转（否则为退化偏差） |
| 噪音注入测试 | `run_noise_test()` | 对数值输入加 ±5% 高斯噪音，检验方向是否稳定（过拟合检测） |
| 跨周期锚定 | `run_cross_cycle_test()` | 禁止通用宏观叙事，检验结论是否依赖板块特异性因子 |
| 综合审计 | `run_full_lcs_audit()` | 加权组合（镜像50%+噪音30%+跨周期20%），输出 `LCSResult` |

权重设计理由：镜像测试是最强信号（逻辑退化检测），噪音测试是稳健性检测，跨周期锚定检测 alpha 来源真实性。

统计门（Block Bootstrap 置换检验）✅ 已实装（§九）：
- `compute_permutation_p_value()` 返回 `PermutationResult` dataclass，三状态输出（insufficient_data / not_significant / significant）
- `bonferroni_adjusted_threshold()` 支持 Bonferroni / Romano-Wolf 两种多重检验校正
- `_MIN_N_PERMUTATION = 100`：样本不足时门控拦截，不产生噪声判断
- Admin 面板已集成置换检验报告与进度可视化（见 §九）

### 5.5 关键设计决策

**1. LCS 不阻断验证，只阻断写回**
LCS 失败的决策仍会正常接受 `accuracy_score`，仍会出现在 Clean Zone 统计中。
区别在于：其结果不会污染 QuantPatternLog / LearningLog。这保证了评估的完整性。

**2. NULL 不等于失败**
`lcs_passed = NULL` 表示 LCS 尚未运行（比如验证时 model 不可用）。
门控逻辑是 `is False` 而非 `!= True`，避免误拦截旧记录。

**3. 镜像测试权重最高（50%）**
镜像测试是检测「无论输入什么都给同一结论」的最直接方法。
噪音测试（30%）补充检测过拟合到特定数值。跨周期锚定（20%）评估 alpha 来源质量。

**4. Clean Zone 目前样本为零**
2025-04-01 刚过，暂无 Clean Zone 记录。面板的价值在后续积累真实 out-of-sample 数据后逐渐显现。这正是「测量系统先于数据建立」的正确顺序。

## 六、三重障碍法 (Triple-Barrier Method)

### 6.1 动机

原有固定窗口验证（短期5d / 中期40d / 长期90d）的本质缺陷：同一个方向正确的信号，若在第 38 天已经兑现并回吐，在第 40 天的固定点位打分会得到错误的低分。三重障碍法把「成立条件」从「第 N 天的点位收益」改为「路径上哪条边界先被触碰」，是对信号兑现机制更诚实的描述。

### 6.2 参数设计（经济逻辑锚定，样本外永不调整）

```
止盈（TP） = 1.0 × 水平 σ
止损（SL） = 0.7 × 水平 σ      ← 非对称：止损更紧，反映不利行情切断更快
时间障碍   = 2 × 信号半衰期

水平 σ = 年化波动率 × √(交易日数 / 252)
年化波动率 = 决策日前 252 个交易日的历史实现波动率（决策日前数据，无未来信息）

两个 horizon 对应的时间障碍（短期预测已移除，见 §七）：
  季度(3个月)：2 × 45 天 = 90 日历天（约 63 个交易日）
  半年(6个月)：2 × 90 天 = 180 日历天（约 126 个交易日）
```

**关键设计决策**：`_TB_TP_MULT = 1.0`、`_TB_SL_MULT = 0.7` 作为模块级常量硬编码，注释明确标注「DO NOT tune」。调整这两个参数等于把窗口选择的过拟合问题换了个形式继续存在。

### 6.3 评分规则

| 触碰边界 | 方向：超配 | 方向：低配 | 方向：标配/拦截 |
|---|---|---|---|
| **止盈（tp）** | 1.0（完全正确） | 0.0（做空被逼空） | 回退到原始打分逻辑 |
| **止损（sl）** | 0.0（方向明确错误） | 1.0（做空方向正确） | 回退到原始打分逻辑 |
| **时间障碍** | `score_accuracy(方向, 时间点收益)` | 同左 | 同左 |

时间障碍兜底用原始 `score_accuracy()` 函数，保持向下兼容。

## 七、Horizon 系统重构：中期/长期 → 季度(3个月)/半年(6个月)

### 7.1 动机：命名与验证窗口的完全对应

原系统存在根本性的定义不一致：

| 原标签 | 声明窗口 | 实际验证时间障碍 | 偏差 |
|---|---|---|---|
| 中期(3-6m) | 3–6 个月 | 90 天（区间下限） | 最多差3个月 |
| 长期(1y+) | 1 年以上 | 180 天（仅半年） | 差6个月以上 |

这种「自我欺骗」会让系统学到的规律与实际验证机制脱钩，是数据严谨性的根本性缺陷。

### 7.2 新 Horizon 体系


## 八、信息层混淆分析与学习回路污染修复

### 8.1 问题识别

从顶级量化分析视角对系统进行了完整的**信息源 × Agent 能力 × 预测 Horizon** 三维匹配审查，识别出以下核心矛盾：

**三层信息被平铺进同一 prompt，LLM 自由定权：**

| 信息层 | 数据 | 自然信号半衰期 | 被注入的预测 Horizon |
|---|---|---|---|
| Tactical 层 | 新闻（days-old）、RSI(14)、Bollinger(20)、VIX spot | 3–21 天 | 季度/半年 ❌ |
| Momentum 层 | mom_1m | 1–3 个月 | 季度（弱）|
| Structural 层 | mom_3m、mom_6m、FRED macro、yield spread | 3–12 个月 | 季度/半年 ✅ |

LLM 会优先锚定最具体、最生动的信息（新闻标题、RSI数值），而不是枯燥的 FRED 百分比。这导致系统实际上是在**用 7 天信号做 90-180 天预测**。

**关键结论——为什么对于本系统这不只是理论问题：**

主要训练来源是 2015-2023 年回测数据，而这段历史在 LLM 训练数据中。LLM 对回测期间的方向判断大量正确，但正确的原因是**前置知识（foreknowledge）**而非信号本身。问题在于 XAI 归因块——LLM 在输出正确方向时，会同时写 `signal_drivers: RSI超卖+Fed转向+动量背离`。三重障碍验证通过后，LearningLog / QuantPatternLog 将这个**虚假相关**当成真实规律存储。

污染后果：Clean Zone（2025-04-01 后）开始真正预测时，LLM 没有了前置知识，但 prompt 里被注入了基于 foreknowledge 作弊积累的「历史学习模式」，例如「宽松制度·科技·RSI超卖 → 季度超配历史胜率78%」。这个 78% 是虚假的，但系统会持续强化它。数据越积累，偏差越根深蒂固。

### 8.2 修复原则

不修改 prompt 输入端（LLM 对信号半衰期有一定自知能力，收益边际小），而是精准修复**学习写回端**的三条污染路径：

```
污染路径 1：  _feedback_routing_weights()
  新闻类别在 90/180 天验证后被强化 → 与验证窗口时序不匹配

污染路径 2：  _update_quant_pattern() → QuantPatternLog
  RSI 状态的胜率统计混合了季度/半年决策 → horizon 不明的模糊先验

污染路径 3：  _compress_skill() → SkillLibrary
  行为指令压缩时可能将 RSI/技术信号编码进触发条件
```

### 8.3 修复详情

#### 路径 1：`_feedback_routing_weights()` — 新闻路由权重写入门控

在函数入口增加 Horizon Gate（位于 train/test 门控之前）：

```python
_h_raw = (dec.horizon or "季度(3个月)").strip()[:1]
if _h_raw in ("季", "半", "中", "长"):
    logger.debug("NewsRoutingWeight write suppressed (horizon mismatch): ...")
    return
```

效果：由于系统现有全部 horizon 均为季度/半年，新闻路由权重写入被永久停止，退回为静态默认权重。这是正确的：比起用 foreknowledge 时代的虚假相关训练出来的动态权重，静态默认值反而更干净。

#### 路径 2：`_state_fingerprint()` / `_update_quant_pattern()` / `get_quant_pattern_context()` — Horizon 标签隔离

**`_state_fingerprint()` 新增 `horizon_key` 参数：**

```python
# 旧：动量:上升|RSI:超卖|波动:高
# 新：动量:上升|RSI:超卖|波动:高|H:季
def _state_fingerprint(state_vector: dict, horizon_key: str = "季") -> str:
```

旧 QuantPatternLog 行（无 `|H:` 后缀）将不再被新查询匹配，成为不可读取的遗留数据。这是设计意图：horizon 混合的历史模式不是可靠先验。

**`_update_quant_pattern()`** 从 `dec.horizon` 提取 horizon_key 传入指纹，每条新记录从此精确属于「季度模式」或「半年模式」。

**`get_quant_pattern_context()`** 改为同时查询两个 horizon 指纹，返回带标签的双块输出：

```
[QUANT_PATTERN_PRIOR · 季度(3个月)]
历史状态：动量:上升|RSI:超卖|波动:高 × 宽松制度  样本=14
超配 72%（n=11）
  ✓ 季度下超配历史胜率良好（72%），量化信号与配置方向一致。
[/QUANT_PATTERN_PRIOR · 季度(3个月)]

[QUANT_PATTERN_PRIOR · 半年(6个月)]
历史状态：动量:上升|RSI:超卖|波动:高 × 宽松制度  样本=8
超配 55%（n=7）
[/QUANT_PATTERN_PRIOR · 半年(6个月)]
```

AI 可直接看到同一市场状态在不同 horizon 下的准确率差异，辅助其 horizon 选择和置信度校准。

#### 路径 3：`_compress_skill()` — 行为指令中禁止短期信号触发条件

在压缩 prompt 增加第 5 条规则（理由内联，让 LLM 理解而非盲目遵守）：

> 【严禁】将 RSI、Bollinger Band、短期技术反弹、超买/超卖等信号半衰期 <30 天的技术指标作为季度/半年行为规则的触发条件——这类信号的预测窗口与验证窗口不匹配，写入规则会污染未来决策的信号归因。触发条件只能基于宏观制度、动量趋势、基本面变化。

### 8.4 未修改的部分及原因

| 组件 | 决策 | 理由 |
|---|---|---|
| Prompt 输入端（RSI 仍被注入） | 保留 | LLM 对短期信号有自知能力，且 RSI 对入场时机（非方向）仍有参考价值 |
| `_update_quant_pattern()` 的 RSI 维度 | 保留 | 这是在记录「我们在 RSI=X 状态下的历史准确率」，是合法的经验统计，而非声称 RSI 是因果驱动 |
| `run_meta_agent_analysis()` 的 signal_bias 检测 | 保留 | 已正确检测「过度依赖技术信号时准确率是否下降」，是检测机制而非污染源 |
| `_check_horizon_mismatch()` 中 r5 的移除 | 已在 §七 完成 | 5日窗口无对应 horizon，mismatch 比较无意义 |

## 九、Block Bootstrap 置换检验实装

### 9.1 动机

在 §三 和 §五 中，置换检验作为"待实现"的统计工具列出。本次完成实装，核心考量：

1. **数据积累速度**：系统以季度为最短决策周期，16 板块，每板块验证需等待 90-180 天。依靠实时积累，每桶（板块 × regime）达到有效统计量需要数年。
2. **历史数据训练为主**：主攻方向是利用 2015-2023 年回测数据进行训练。在单次系统性回测运行后，每桶样本可达 100 条以上，置换检验可立即激活。
3. **先基础设施，后激活**：门控阈值 `_MIN_N_PERMUTATION = 100` 使检验在样本不足时静默返回 `insufficient_data`，不影响系统运行；一旦样本充足，自动激活，无需代码修改。

### 9.2 门控阈值选择依据

```
理论背景：单比例双侧 z 检验，H0: accuracy = 50%，α = 0.05

n  =  50 → 需 ≥ 69.4% 准确率才能通过  ← 太严格：真实 65% 信号被误判为不显著
n  = 100 → 需 ≥ 63.8% 准确率才能通过  ← 可检测 65%+ 信号  ✓
n  = 200 → 需 ≥ 59.7% 准确率才能通过  ← 最稳健，但需更多数据积累

_MIN_N_PERMUTATION = 100
```

n=50 的问题不是"风险大"，而是检验力不足：真实存在的信号（65% 准确率）会因为统计功效不够而被误判为不显著，整个检验毫无意义。n=100 是"能看见真实 alpha"的最低门槛。

### 9.3 新数据结构：`PermutationResult`

```python
@dataclass
class PermutationResult:
    status: str              # "insufficient_data" | "not_significant" | "significant"
    p_value: Optional[float] # None when insufficient_data
    observed_accuracy: float
    n_samples: int
    n_needed: int            # = _MIN_N_PERMUTATION
    adjusted_threshold: float
    sector: str = ""
    regime: str = ""

    @property
    def passed(self) -> bool:
        return self.status == "significant"

    @property
    def progress_pct(self) -> float:
        return min(1.0, self.n_samples / max(1, self.n_needed))
```

三状态设计的关键：`insufficient_data` 与 `not_significant` 是完全不同的含义。前者是"还没有足够数据做判断"，后者是"数据足够但信号不显著"。混淆这两种状态会导致错误的系统信心（或过度悲观）。

### 9.4 `engine/lcs.py` 变更

**`compute_permutation_p_value()` 重构为返回 `PermutationResult`：**

```
入参：accuracy_list (训练集已验证决策的准确率列表), n_permutations=10_000 (从 1,000 提升)
出参：PermutationResult

门控逻辑：
  n < _MIN_N_PERMUTATION → status="insufficient_data", p_value=None
  n >= 100 → 块自举（块长4 = 季度），10,000 次重排
            → Romano-Wolf 校正阈值
            → p ≤ adjusted_threshold: "significant"，否则 "not_significant"
```

**默认 `n_permutations` 从 1,000 → 10,000**：样本量 100 时，1,000 次重排 p 值精度仅到小数点后两位，10,000 次才能稳定在小数点后三位。

### 9.5 `engine/memory.py` 新增 `get_permutation_report()`

```python
def get_permutation_report(n_permutations: int = 10_000) -> list[dict]:
```

逻辑：
1. 查询训练集内 (`decision_date < TRAIN_TEST_CUTOFF`) 已验证 (`accuracy_score IS NOT NULL`) 的决策
2. 按 `(sector × regime)` 分组
3. 每组调用 `compute_permutation_p_value()`
4. 返回每组的 `PermutationResult` 序列化字典（status / p_value / n / n_needed / progress_pct / sector / regime / observed_accuracy）

### 9.6 `pages/admin.py` Admin 面板集成

在「Learn」Tab 新增「Signal Significance · Block Bootstrap Permutation」模块：

| UI 元素 | 功能 |
|---|---|
| 快速模式切换 | 1,000 次（快，约 1s）vs 10,000 次（精确，约 10s） |
| 运行按钮 | 调用 `get_permutation_report()`，spinner 显示进度 |
| 三列汇总 | 显著信号数 / 不显著数 / 数据不足数（按颜色区分） |
| 数据不足组 | 进度条显示 `n / n_needed`，直观反映距激活还差多少样本 |
| 显著/不显著组 | 展示 p 值、实测准确率、样本量 |

颜色编码：
- `significant`：绿色 `#10B981`
- `not_significant`：琥珀 `#F59E0B`
- `insufficient_data`：中性灰 `#6B7280`（进度条蓝色）

## 十、系统设计原则：历史数据训练策略与风险控制架构

### 10.1 为什么历史数据是主攻方向

| 维度 | 实时积累 | 历史回测 |
|---|---|---|
| 效率 | 每季度 1 条决策/板块 | 单次运行覆盖 2015-2023 全部 8 年 |
| 样本速度 | n=100/桶 需 ≥ 6 年 | n=100/桶 可在数小时内完成 |
| 结构覆盖 | 局限于当前 regime | 覆盖低利率/加息/危机等多个 regime |

效率优势约 10-100 倍。代价是：历史数据的数据风险是实时数据的 10 倍以上——LLM 对 2015-2023 的市场走向有前置知识，每一个"正确"预测都可能是 foreknowledge 而非信号本身。

### 10.2 为什么所有质量控制都是承重墙

正因为历史训练效率最高、风险也最大，本系统积累的所有质量控制机制都是**承重墙**而非装饰——移除任何一个都会导致历史训练的结果在 Clean Zone 失效：

```
TRAIN_TEST_CUTOFF (2023-01-01)
  → 防止测试集结果回流训练，保护 out-of-sample 评估的有效性

CLEAN_ZONE_START (2025-04-01)
  → 区分"受 LLM foreknowledge 污染"与"真实 out-of-sample"

三重障碍法
  → 防止信号在验证窗口终点已回吐但被错误打高分

LCS 质量门
  → 防止 foreknowledge 驱动的逻辑退化（无论输入什么都给相同结论）

Horizon 标签指纹 (|H:季/半)
  → 防止不同验证窗口的胜率在 QuantPatternLog 中混合

新闻路由权重门控
  → 防止短期新闻相关性被错误强化为季度/半年预测规则

置换检验门控 (_MIN_N_PERMUTATION = 100)
  → 防止小样本噪声被误判为显著 alpha

SkillLibrary 压缩规则 Rule 5
  → 防止 foreknowledge 时代虚假相关的技术信号被编码进永久行为指令
```

历史训练不是"可以做"的选项，它是系统达到有效运行状态的唯一合理路径。但这条路上的每一个质量控制都不可省略。

### 10.2.1 低样本噪声守卫（Soft Guard）

`_update_quant_pattern()` 在每条验证完成后立即写入 QuantPatternLog（不阻塞）。在阶段 0 历史批量训练完成（基线 n ≥ 100）之前，若提前跑实时决策，前几十条更新的先验信噪比极低——n=10 时一条错误决策会将胜率从 60% 拉到 54.5%，这个数字会被注入下一个 prompt。

为此在写入完成后增加软检查：

```python
_BUCKET_NOISE_THRESHOLD = 20
if row.total_count < _BUCKET_NOISE_THRESHOLD:
    logger.warning(
        "QuantPatternLog low-n write: fingerprint=%s regime=%s direction=%s "
        "n=%d (<%d) — prior is noisy; run Phase-0 historical batch training ...",
        ...
    )
```

**不阻断写入**（保持数据完整性），**只输出 WARNING 级日志**，提示运维人员当前桶尚未建立可靠基线。管理员在 Admin 面板的日志流中可见此告警。

### 10.3 已知局限性（设计层面）

以下局限性属于架构约束，已知但当前不予修改，记录备查：

| 局限性 | 描述 | 影响 |
|---|---|---|
| FRED 修正偏差 | FRED 宏观数据存在历史修正，回测中使用的是修正后数值，而非当时的发布值 | 宏观特征的训练样本比真实情况偏乐观 |
| LLM 前置知识量未定量 | 无法精确测量 foreknowledge 对每条历史预测的贡献比例 | Clean Zone 前的准确率无法信任，但污染程度未知 |
| QuantPatternLog walk-forward 语义 | 历史训练写入的模式先验（尤其是 2015-2019 数据）在 walk-forward 严格意义上不是 out-of-sample | 胜率先验存在轻微过度乐观，需以 Clean Zone 实际表现校正 |
| 单 LLM 架构 | 所有 Agent 共享同一模型权重，"红队"和"蓝队"本质上是同一模型 | 辩论质量依赖 prompt 工程而非真正独立视角 |

## 十一、Agent 节点接入 Alpha Memory + 证据链剥离

### 11.1 变更背景

`agent.py` 的 `red_team_node` 此前每次运行都是白纸一张——不知道这个板块在当前 regime 下历史胜率多少，也不知道系统过去在类似场景下的偏差方向。同时，红队默认可以直接复用量化系统已经给出的正面指标（3M动量为正、波动率在控等）作为放行依据，失去了真正独立反驳的价值。

两个问题，一次修复。

### 11.2 `engine/memory.py` — `get_historical_context()` 改为 cross-tab 可查

原函数签名要求必须传 `tab_type`，且以 `tab_type == tab_type` 过滤，导致 agent.py 无法查询 tabs.py 写入的决策记录：

```python
# 旧：tab_type 必传，强制过滤
def get_historical_context(tab_type: str, ...) -> str:
    q = q.filter(DecisionLog.tab_type == tab_type, ...)

# 新：tab_type="" 或省略时跳过过滤，跨 tab 查询
def get_historical_context(tab_type: str = "", ...) -> str:
    if tab_type:
        q = q.filter(DecisionLog.tab_type == tab_type)
```

向后兼容：现有调用方均传非空 `tab_type`，行为不变。

### 11.3 `engine/agent.py` 三处变更

**1. `AgentState` 新增 `macro_regime` 字段：**

```python
macro_regime: str   # 结构化制度标签，如 "宽松"/"收紧"/"高波动收缩"
```

调用方在启动 agent 时传入当前 regime，与 `macro_context`（非结构化文字）并列。Alpha Memory 的 sector × regime 查询依赖此字段。

**2. Alpha Memory 注入（数据为空时零影响）：**

```python
alpha_ctx = get_historical_context(
    tab_type="",          # cross-tab
    sector_name=asset_name,
    macro_regime=state.get("macro_regime", ""),
)
alpha_block = f"\n{alpha_ctx}\n" if alpha_ctx else ""
```

`get_historical_context()` 在 DB 为空时返回 `""`，对 prompt 零影响。Phase-0 历史训练完成后自动激活，无需再改代码。

**3. 证据链剥离（Blacklist 机制）：**

从 `quant_results` 中提取量化系统已经确认的主要利好论据，在红队 prompt 中明确禁止以这些证据作为放行依据：

```python
blue_drivers = []
if q.get("mom_3m") is not None:
    blue_drivers.append(f"3M动量为{'正' if q['mom_3m'] > 0 else '负'}（{q['mom_3m']:+.1%}）")
if q.get("a_ret", 0) > 0:
    blue_drivers.append(f"年化收益率为正（{q['a_ret']:+.1%}）")
if q.get("p_noise", 1) < 0.3:
    blue_drivers.append(f"P-hacking风险低（{q['p_noise']:.1%}）")
if q.get("a_vol", 0) < 0.2:
    blue_drivers.append(f"年化波动率在控（{q['a_vol']:.1%}）")
# → 注入 prompt：禁止将上述指标作为放行依据，必须提出独立反驳视角
```

效果：红队被强制从已知证据路径之外寻找风险，而不是礼貌地"补充"量化系统的结论。这是上述辩论架构分析中唯一确认高价值的机制。

### 11.4 设计决策

**只接 `red_team_node`，不接其他三个节点：**

| 节点 | 是否接入 | 理由 |
|---|---|---|
| `red_team_node` | ✅ | 审计核心，历史胜率先验直接改变审计力度 |
| `reflection_node` | ❌ | 只从排行榜找替代，不做方向判断 |
| `technical_audit_node` | ❌ | 纯技术指标计算，与 Alpha Memory 无关 |
| `translator_node` | ❌ | 格式转译，不做决策 |

**数据量小时不是不接的理由：** 函数在 DB 为空时返回 `""`，连接的代价是两次毫秒级空查询。不连则需要 Phase-0 完成后再补接线——基础设施先于数据建立的原则同样适用于此。

## 十四、未来技术路线图

以下是从量化方法论和系统工程双重视角评估后，认为值得在后续版本实现的技术方向，按优先级排序：

### 优先级 HIGH（影响系统可用性）

**Phase-0 历史批量训练执行**
当前所有学习机制的基础设施已就绪，但 QuantPatternLog / SkillLibrary 仍为空。最高优先级的不是新功能，而是用 2015-2023 历史回测数据把现有系统跑通一次，验证整条学习链路端到端有效。

**Agent 节点完整接入 Alpha Memory**
`red_team_node` 已接入（§十一）。下一步：`reflection_node` 在选替代板块时，也应能查询该候选板块的历史胜率先验，避免推荐一个系统历史上从未有效的板块作为替代。

**Walk-forward 回测验证框架**
目前 `run_walk_forward_backtest()` 在 history.py 中存在，但没有标准化的性能报告（Sharpe / Max Drawdown / Hit Rate 随时间演化）。需要一个端到端的 walk-forward 验证 pipeline，将 Phase-0 训练质量可视化。

### 优先级 MEDIUM（改善预测质量）

**James-Stein 收缩估计**
QuantPatternLog 中小样本桶（n < 30）的胜率估计噪声高。JS 收缩将各桶向全局先验（跨板块平均胜率）收缩，尤其对冷门板块有显著改善。前提：置换检验已确认系统有真实 alpha。

**对数收益率全面替换**
当前动量计算和收益统计混用简单收益率和对数收益率。对数收益率的时间可加性在多期动量计算（mom_3m, mom_6m）中更严谨，且消除正偏斜导致的小样本过度乐观。影响面广，需统一改造 `engine/quant.py`。

**Regime 变迁检测**
`_infer_macro_regime()` 目前只用 VIX 单因子分类。更稳健的做法是叠加 yield curve slope + credit spread，构建多因子 regime 标签，减少 VIX 峰值期的误分类。QuantPatternLog 的桶划分精度直接依赖 regime 标签质量。

**SkillLibrary 年度重压缩机制**
当前 SkillLibrary 只在单桶样本达到 `_SKILL_REFRESH_EVERY` 时触发更新。应增加年度全量重压缩：在每年年初，对所有桶用最近 12 个月数据重新生成行为指令，清除过时规律，同时检测与历史版本的语义漂移。

**探索覆盖率追踪（Exploration Coverage Tracking）**
系统有 16 板块 × 4 制度 = 64 个 (sector × regime) 桶。Phase-0 历史训练结束后，各桶的样本量极不均衡——历史上牛市×科技的决策远多于高波动收缩×能源。未被充分探索的桶在 QuantPatternLog 中先验置信度低，但系统目前无法感知这一状态。

建议实装一个覆盖率矩阵：
```
Priority(cell) = 0.5 × Novelty(cell) + 0.3 × HistFailRate(cell) + 0.2 × NeedMore(cell)
# Novelty: 1 - n/n_max（样本稀疏度）
# HistFailRate: 该桶历史失败率（高失败桶优先补充诊断数据）
# NeedMore: 距置换检验激活门槛 (n_needed - n) / n_needed
```

Admin 面板热力图可视化 64 个桶的覆盖状态，为 Phase-0 历史回测的执行顺序提供优先级指引。灵感来源：BrainAlpha 的 `Priority(c,a)` 函数（Novelty + ExpectedYield + AgentAlignment 加权）。

### 优先级 LOW（架构完整性）

**FRED 实时发布值接入**
当前宏观特征使用 FRED 修正后数据，与真实决策时可见的初始发布值存在偏差（即 §十中记录的局限性）。接入 FRED API 的 `vintage` 参数，记录每个决策日期实际可见的发布值，而非事后修正值。

**多模型辩论架构（条件性）**
仅在 Clean Zone 样本 ≥ 30 条、且单模型辩论的 LCS 一致性得分持续偏高（> 0.85）时考虑引入异构模型（Claude 红队 + Gemini 蓝队）。证据链剥离机制（§十一）已能拿到大部分收益，双模型调用的 API 成本需有实证依据支撑。

**SpilloverWeight 学习层激活**
devlog_2026-04-09 中设计的板块传导权重学习机制（`SpilloverWeight` 表）已有 schema，但激活条件（每对板块同期验证决策 ≥ 15 条）尚未达到。Phase-0 完成后应自动触发，需在 Admin 面板增加传导热力图可视化。

## 十四、预测失败归因系统（Failure Mode Taxonomy）

### 14.1 设计背景与灵感来源

BrainAlpha（2025）在代码表达型 Alpha 研究中定义了六类失败模式（FM-1 逻辑谬误、FM-2 数据获取失败、FM-3 样本偏差……）用于指导 LLM 生成代码的后验分析。其核心思路具有迁移价值：**不只记录"预测错了"，而是诊断"因为什么错了"**，才能在后续训练和 prompt 设计中有针对性地干预。

本系统的预测失败来源与代码生成不同——错误根源在自然语言推理层而非代码执行层——因此对 BrainAlpha 的分类框架进行了重新映射：

| 本系统 | 对标 BrainAlpha | 核心机制 |
|---|---|---|
| FM-A（逻辑退化） | FM-1 逻辑谬误 | LCS 镜像检测 |
| FM-B（过度自信） | FM-3 样本偏差 | confidence_score ≥ 85 |
| FM-C（信号污染） | FM-4 前瞻偏差 | 短期信号词汇检测 |
| FM-D（制度误分类） | FM-5 分布偏移 | 同板块×制度连续失败 |

### 14.2 四类失败模式定义

| 代号 | 名称 | 触发条件 | 含义 |
|---|---|---|---|
| FM-A | 逻辑退化 | `lcs_mirror_passed is False` | 红队和蓝队在镜像测试下给出一致结论，辩论失去独立性——两个 agent 实际上是在互相确认而非独立质疑 |
| FM-B | 过度自信 | `confidence_score ≥ 85` | 系统以高置信度做出了最终错误的预测，表明校准机制失效 |
| FM-C | 信号污染 | `signal_attribution` 包含短期技术信号关键词 | 短期信号（RSI/Bollinger/MACD/KDJ/超买/超卖）主导了季度/半年预测的归因，信号半衰期与预测窗口不匹配 |
| FM-D | 制度误分类 | 同板块×制度 2 次连续失败 | 系统可能在错误的制度标签下进行了预测，或宏观制度发生了转变而系统未及时识别 |

分类优先级：FM-A → FM-B → FM-C → FM-D（依次检查，取第一个匹配）。未能覆盖的失败记录为 `unlabelled`（空字符串）。

### 14.3 实装：`engine/memory.py`

**新增字段（`DecisionLog` ORM）：**

```python
failure_mode = Column(String(8), nullable=True)
# 取值：""(NULL) | "FM-A" | "FM-B" | "FM-C" | "FM-D"
```

**数据库迁移（`_migrate_db()`）：**

```python
dl_extra_columns = [
    ...
    ("failure_mode", "VARCHAR(8)"),
]
```

**`_SHORT_TERM_SIGNAL_MARKERS` 常量（FM-C 用）：**

```python
_SHORT_TERM_SIGNAL_MARKERS = (
    "RSI", "Bollinger", "MACD", "KDJ",
    "超买", "超卖", "均线", "布林",
)
```

**`_classify_failure_mode(dec, session) -> str`：**

```python
def _classify_failure_mode(dec: DecisionLog, session) -> str:
    if dec.accuracy_score is None or dec.accuracy_score >= 0.5:
        return ""   # 未失败，不归因
    if dec.lcs_mirror_passed is False:
        return "FM-A"
    if (dec.confidence_score or 0) >= 85:
        return "FM-B"
    # FM-C：检查 signal_attribution 是否含短期技术信号词
    drivers = ""
    if isinstance(dec.signal_attribution, dict):
        drivers = dec.signal_attribution.get("drivers", "")
    elif isinstance(dec.signal_attribution, str):
        drivers = dec.signal_attribution
    if any(m in drivers for m in _SHORT_TERM_SIGNAL_MARKERS):
        return "FM-C"
    # FM-D：同板块×制度最近 2 条记录均已失败
    prior_fails = (
        session.query(DecisionLog)
        .filter(
            DecisionLog.sector_name == dec.sector_name,
            DecisionLog.macro_regime == dec.macro_regime,
            DecisionLog.id != dec.id,
            DecisionLog.accuracy_score < 0.5,
        )
        .order_by(DecisionLog.id.desc())
        .limit(2)
        .all()
    )
    if len(prior_fails) >= 2:
        return "FM-D"
    return ""
```

**写回时机**（在 `verify_pending_decisions()` 中，`_run_lcs_on_decision()` 之后）：

```python
dec.failure_mode = _classify_failure_mode(dec, session)
if dec.failure_mode:
    logger.info(
        "FailureMode: id=%d sector=%s regime=%s → %s",
        dec.id, dec.sector_name, dec.macro_regime, dec.failure_mode,
    )
```

**`get_failure_mode_stats() -> dict`：**

聚合查询所有已验证决策（`accuracy_score IS NOT NULL`），按失败模式分组，返回：

```python
{
    "total_verified": int,
    "total_failed": int,
    "by_mode": {"FM-A": int, "FM-B": int, "FM-C": int, "FM-D": int},
    "fm_labels": {"FM-A": "逻辑退化", ...},
    "unlabelled": int,        # 失败但未被任何 FM 捕获
    "by_sector": dict,        # 每个板块的失败模式分布
}
```

### 14.5 手动规则的局限性与后续优化方向

当前实现是**规则代理**，适合数据量小的初期阶段：

| 维度 | 规则层（当前） | AI 分析层（后续） |
|---|---|---|
| FM-A（LCS 镜像） | 机械准确，无需 AI | — |
| FM-B（过度自信） | 机械准确，无需 AI | — |
| FM-C（信号污染） | 关键词命中，漏判率高（提及≠主驱动） | LLM 分析辩论实录，判断信号在推理链中的权重 |
| FM-D（制度误分类） | 连续失败为结果观察，非原因诊断 | LLM 对比决策日的 regime 标签与实际市场制度特征 |

**当前只实装规则层，未实装 LLM post-mortem。**

评估结论：post-mortem 与已有机制重叠，无增量价值——
- 个案 LLM 分析已由 `_ai_reflection()` 覆盖（写入 `dec.reflection`）
- 系统性偏差发现已由 `run_meta_agent_analysis()` 覆盖（统计聚合路径，不依赖 FM 标签）
- FM 标签目前只用于 Admin 面板可视化，没有下游自动行为消费它

每条失败额外一次 LLM 调用，换来的只是饼图里稍微更准确的分类数字，成本收益不对称。**规则层够用。**

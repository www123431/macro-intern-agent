## 二、Key Pool Manager 功能增强

**文件**：`pages/key_manager.py`、`engine/key_pool.py`

### 2.1 手动强制切换 Key

新增 `KeyPoolManager.force_switch_to(label)` 公开方法，直接修改 `_idx` 指针，不受 Key 状态（exhausted / halted）限制，作为自动轮转机制的冗余备份。

```python
def force_switch_to(self, label: str) -> None:
    with self._lock:
        for idx, (lbl, _) in enumerate(self._keys):
            if lbl == label:
                self._idx = idx
                return
```

UI 层在 Key 配置管理区新增 Section 3「手动切换 Key · 冗余备份」，selectbox 列出所有 Key（带状态图标），切换后提示生效。自动切换逻辑（`_rotate_key`）完全不受影响，手动切换只改变起点。

### 2.2 Key 健康诊断

在 Pool Overview 汇总卡片下方新增两条诊断提示：

- **冷启动配额异常（黄色）**：`today_calls == 0` 且已出现错误或处于 exhausted/halted 状态，说明 Key 在未成功调用前就已失效，建议手动切换至备用 Key。
- **今日未使用提示（蓝色）**：`today_calls == 0` 且状态正常，仅作信息展示。

诊断时序跟随 Gemini 配额日（PT 午夜 = SGT 16:00），而非系统本地时间。实现方式：在 `get_all_stats()` 内部先调用 `_reset_daily_counters_if_needed()`，确保每次读取统计数据前完成 Gemini 配额日同步，即使 App 跨 SGT 16:00 持续运行也不会读到过期计数。

手动切换时若选中的 Key 已是当前 Key，使用 `st.empty()` 占位符显示提示 3 秒后自动清除，避免残留 UI 元素。

**文件**：`engine/memory.py`、`ui/tabs.py`

### 4.1 背景

中长期投资建议（季度/半年）基于发布时的宏观假设，当假设失效（如 VIX 突破阈值、宏观制度切换）时需要修订。现有系统已有失效条件记录（`invalidation_conditions`）和 Tab2 失效条件监控，但缺少从检测到修订的完整闭环。

### 4.2 数据库 Schema 变更

`DecisionLog` 新增三个字段：

| 字段 | 类型 | 用途 |
|---|---|---|
| `superseded` | BOOLEAN DEFAULT 0 | 标记该决策已被修订替代 |
| `parent_decision_id` | INTEGER | 指向被修订的原始决策 ID |
| `revision_reason` | TEXT | 触发修订的具体原因 |

```python

    # 获取单条决策的关键字段，用于修订分析时注入原始判断上下文
```

### 4.4 性能统计保护

`verify_pending_decisions` 和 `get_clean_zone_stats` 均新增 superseded 过滤条件，确保被替代的旧版决策不污染胜率、准确率等绩效统计。`_check_invalidation_alerts` 同步过滤 superseded 决策，避免已修订的条件重复触发。

### 4.5 修订分析管道

`_run_sector_analysis` 新增 `parent_decision_id` 和 `revision_reason` 参数。修订模式下：

1. 从 Alpha Memory 拉取原始决策（方向、逻辑摘要、失效条件）
2. 将以下修订上下文前置注入 `hist_ctx`，传递给 LLM 辩论系统：

```

```

3. 新决策保存时携带 `parent_decision_id`
4. 保存成功后调用 `supersede_decision` 标记原始决策

### 4.6 Tab2 失效条件监控 UI 升级

每条已触发的失效条件卡片新增「修订」按钮，点击后直接在 Tab2 内发起修订分析，无需跳转 Tab3。

### 4.7 自动触发系统

新增自动修订开关，采用 **arm-then-fire** 模式：

- 默认关闭，用户需手动开启
- 开启时触发 **10 秒倒计时警告**（提示配额消耗和 Alpha Memory 写入副作用），到期后自动激活
- 激活后每次 Tab2 渲染时检查触发条件，对满足条件的决策自动发起修订分析
- **24 小时/板块冷却机制**（持久化至 DB，见下文补充）防止同一条件在多次页面刷新中反复触发

Streamlit 技术约束说明：「自动」触发依赖页面刷新（rerun），非真后台守护进程。在用户停留于 Tab2 并发生交互时执行，适合当前使用场景。

## 六、手动修订二步确认 UX

**文件**：`ui/tabs.py`

**改动**：将 Tab2 失效条件卡片上的「修订」单击直接触发改为两步确认流程。

**原因**：修订分析会消耗 Gemini 配额并写入 Alpha Memory，误触成本较高。加入确认步骤可防止误操作，同时允许用户在系统预填原因的基础上补充背景，提升修订质量。

**实现**：

```
第一步：点击「修订」
  → session_state["revise_confirm_{id}"] = True
  → 按钮切换为「取消」，支持放弃操作

第二步：展示确认面板（带边框 container）
  → 系统预填自动检测到的触发原因（可直接使用）
  → text_area 可补充背景说明
  → 「确认修订 ▶」(primary) 按钮执行分析
```

每条决策独立使用四个 session_state key（`revise_`、`revise_cancel_`、`revise_reason_`、`revise_submit_` + `decision_id`），多条同时触发时互不干扰。取消后状态干净清除，不留残留。

## 七、历史回测写回隔离 · Alpha Memory 数据完整性修复

**文件**：`engine/memory.py`

### 7.1 背景与问题

经评估，历史回测（`is_backtest=True`）的决策存在根本性的前瞻污染问题：LLM 的预训练语料包含了历史金融事件的叙述和结果，当它分析 2022 年某一历史日期时，它隐式知道该时期发生了什么。这意味着历史回测的产出不是真实的 out-of-sample 预测，而是开卷考试的答案。

**原有设计缺陷**：`TRAIN_TEST_CUTOFF`（2023-01-01）已阻止 2023 年之后的记录写回学习表，但无法阻止 2023 年之前的回测决策写回——而后者恰恰是污染最严重的部分，LLM 对 2020~2022 年的历史记忆最为完整。

### 7.2 修改内容

在四个写回入口统一新增 **backtest contamination gate**，位于原有 test-set gate 之前：

| 函数 | 变更 |
|---|---|
| `_update_quant_pattern` | 新增 `if dec.is_backtest: return` 早退出，阻止写入 `QuantPatternLog` |
| `run_meta_agent_analysis` | 查询过滤新增 `is_backtest == False`，阻止写入 `LearningLog` |
| `maybe_update_skill` | 查询过滤新增 `is_backtest == False`，阻止写入 `SkillLibrary` |
| `get_clean_zone_stats` | 查询过滤新增 `is_backtest == False`，三区间统计只反映 live 决策 |

`_feedback_routing_weights` 已有 horizon gate（当前所有季度/半年决策均被拦截），暂不另加，避免冗余。

### 7.3 设计原则重申

历史回测的合理用途仅限于：
1. **管道验证**：确认数据流、提示词、辩论系统、存储链路跑通
2. **系统调试**：在 Backtest Review Tab 检查记录格式和异常

产出的决策记录永远不写回任何学习表，其验证结果（accuracy_score）只用于调试，不计入三区间绩效统计。

### 7.4 项目定位说明

本系统的本质是**系统化宏观研究辅助框架**，而非传统量化回测系统。LLM 已通过预训练掌握历史金融规律，无需通过回测写回重复学习。Alpha Memory 的价值在于积累**本系统特有的决策脉络**——当前宏观制度下的判断历史、失效条件追踪、修订链——本质是情景记忆（episodic memory），而非通用知识库。

**唯一有效的绩效证据**：Clean Zone（≥ 2025-04-01）live 决策在三重壁垒验证后的胜率。样本积累到 ≥ 30 条之前，任何绩效数字均不具统计稳定性。

### 7.5 定位重塑后 Look-Ahead Bias 的意义变化

重新定位为研究辅助框架后，历史回测层面的 look-ahead bias 对核心结论的影响大幅降低——系统本来就不依靠回测数据说话，污染已被主动承认并隔离。

然而，**Clean Zone live 决策的时间点干净性**因此变得更加关键，而非更不重要。理由如下：

- 回测的污染是已知的、已隔离的，不影响对系统真实性能的判断
- Clean Zone 是系统声称干净的唯一评估区间，若这里存在隐性的 look-ahead（GDELT 执行时间不一致、FRED 数据跨月混用），则系统彻底失去自我验证能力——**以为干净、实际不干净**是比已知污染更危险的状态

因此，缺陷审计中 P1 级别的两个问题（GDELT 截止时间、FRED 发布滞后）在新定位下优先级应上调，是后续最值得优先修复的技术债。

## 八、系统缺陷审计 · 量化视角严格评估

以下缺陷按严重程度分级，P1 为数据完整性问题，P2 为评估逻辑问题，P3 为信号质量问题。所有问题均已识别，暂未修复，作为后续迭代方向记录。

---

### P1 · 数据完整性（影响结论可信度）

**1.1 GDELT 新闻截止时间依赖执行时刻**

`get_gdelt_news()` 以 T-1 为截止，但 GDELT 的索引延迟约为 2~4 小时。若分析在 UTC 06:00 执行（GDELT 约 05:30 发布），部分 T-1 新闻已进入索引；若在 UTC 18:00 执行，T-1 晚间的文章可能尚未收录。这意味着两次相同条件的分析，因执行时间不同，实际摄取的新闻集不同，属于隐性的时间点不一致性。**未设置明确的 UTC 时间戳锁定机制。**

**1.2 FRED 发布滞后假设脆弱**

CPI 固定滞后 35 天、非农 7 天、联邦基金利率 2 天，这些是静态硬编码值，未考虑 BLS/FRED 数据修订周期。`_get_fred_value_on()` 返回截止日前最近一条记录，但无法验证该值是否属于正确的统计月份——若 FRED 在某月数据稀疏（节假日）时，可能跨月混用数据。**无发布时间戳校验机制。**

---

### P2 · 评估逻辑（影响绩效统计可信度）

**2.1 三重壁垒参数缺乏经济依据**

TP 倍数 1.0σ、SL 倍数 0.7σ 在代码注释中被标注为"经济锚定"，实际上是人工设定值，没有对应任何板块层面的实证校准。非对称设计（SL < TP）理论上偏向更频繁触发止损，会系统性压低准确率评分。时间壁垒（90/180 交易日）以固定公式折算，未考虑板块季节性或实际交易日历。这些参数应基于历史数据对每个板块单独拟合，而非全局统一。

**2.2 置信度校准门槛过松**

Meta-Agent 置信度校准检测逻辑：`hc_avg < lc_avg + 0.05`，即高置信组比低置信组准确率高 5% 才触发警告。5% 在统计上接近噪音，无法检测到系统性高估置信度的问题。LLM 输出的置信分数（0~100）通过正则从自由文本中提取，本身就不是校准概率，放入统计分析前需要额外的单调性和稳定性验证。

**2.3 LCS 写回门控产生结构性学习盲区**

写回逻辑：`verified=True AND lcs_passed=True AND date < TRAIN_TEST_CUTOFF`。

这产生了一个结构性问题：
- 决策正确 + LCS 通过 → 写回（但这类决策没有新信息，系统本就擅长）
- 决策错误 + LCS 失败 → 不写回（失败案例被阻断，无法学习改进）
- 决策错误 + LCS 通过 → 写回（这是唯一有学习价值的组合，但也是最稀少的）

系统从失败案例中学习的能力被 LCS 门控大幅削弱，长期可能导致对已掌握模式的过度强化，对未知失败模式的低覆盖。

---

### P3 · 信号质量（影响分析可靠性）

**3.1 辩论系统：红队论证质量无约束**

红队（Red Agent）被强制要求持反向立场，但没有论证质量约束——可以用最弱的反向理由完成任务，不需要找到最强的反驳。例如蓝队基于宏观论证超配，红队可以仅用资金流向的一条新闻作为反驳，而不真正回应宏观逻辑。辩论轮次固定为 2 轮，无动态终止条件，可能在论证未充分展开时就进入仲裁。

**3.2 仲裁节点受蓝队锚定效应影响**

仲裁 Prompt 包含 `state['blue_output'][:800]`，仲裁员在看到红队论证之前已读取蓝队立场。心理学和 LLM 实验均表明，先呈现的信息对最终判断有显著锚定效应。更严格的设计应让仲裁员先独立分析，再参考双方论据。

**3.3 XAI 解析失败静默降级**

`_parse_xai()` 用正则从 LLM 自由文本中提取置信度、信号归因等字段。解析失败时字段静默返回空值，仲裁节点的合并逻辑 `merged.update({k: v for k, v in final_xai.items() if v not in (None, "", {})})` 会自动回退到蓝队的解析值。这意味着：仲裁结果实质上等于蓝队分析的情况可能远比界面展示的更频繁，但无法从记录中区分"真正仲裁"和"静默降级回蓝队"。

**3.4 新闻类别权重从未被验证**

系统存储了 `news_categories_used`（XAI 归因的新闻类别），设计了 `NewsRoutingWeight` 动态权重反馈机制，但因 horizon gate（所有季度/半年决策均触发拦截）实际上所有权重更新均被禁止。系统声称的"新闻权重动态学习"功能目前完全静默。这导致 XAI 中的新闻归因缺乏任何经验验证支撑。

**3.5 状态向量阈值忽略波动率上下文**

动量状态阈值固定为 ±2%：在 VIX=50 的危机环境下，±2% 是日常波动；在 VIX=10 的低波市场，±2% 是极端事件。两种环境下相同的价格变动被映射为相同的状态标签，导致状态向量在高低波动率制度之间缺乏可比性，降低了 `QuantPatternLog` 跨制度聚类的有效性。

**3.6 信用利差代理变量设计粗糙**

HYG-IEF 20 日收益差作为信用利差代理，假设两者呈反向关系。但在系统性 risk-off 阶段，HYG 和 IEF 有时同向下跌（国债被抛售同时信用利差扩大），两者的价格关系并不稳定。更可靠的代理应使用 CDX Investment Grade / High Yield 指数，或直接用 OAS（Option-Adjusted Spread）数据。

**3.7 敏感性测试为死代码**

`_run_sector_analysis` 中调用了 `run_sensitivity_test()`，该函数在整个代码库中未找到定义。返回值被存入 `sensitivity_flag` 字段并写入数据库，但下游无任何消费逻辑。当前所有决策记录中的 `sensitivity_flag` 字段均为无意义占位。

---

### 综合评估

系统架构具备真实价值（对抗辩论、时间隔离、失效条件追踪），但在以下三个维度存在结构性弱点：

| 维度 | 问题 | 当前状态 |
|---|---|---|
| 数据完整性 | GDELT 时间点不一致、FRED 滞后假设脆弱 | 未修复 |
| 评估可信度 | 壁垒参数未校准、LCS 学习盲区 | 未修复 |
| XAI 真实性 | 解析静默降级、新闻权重冻结、敏感性分析缺失 | 未修复 |

Clean Zone 累积足够样本（≥30 条 live 决策）之前，上述问题对实际使用影响有限。优先修复顺序建议：3.3（XAI 静默降级可观测性）→ 2.1（壁垒参数板块校准）→ 1.1（GDELT 执行时间锁定）。

## 九、数据源升级路线图 · 评估与优先级

### 9.1 结论前置

数据源升级有必要，但**现在不是时候**。正确顺序是：验证当前系统是否有效 → 诊断判断质量瓶颈 → 针对性升级。在 Clean Zone 积累足够样本之前盲目升级，是在未验证的基础上加投入。

### 9.2 真正有价值的升级（待 Clean Zone 验证后执行）

**付费新闻数据源（替换 GDELT）**

GDELT 的核心问题不是覆盖面，是时效和时间戳精度。付费数据源（Bloomberg News、Refinitiv Eikon News）提供精确的 UTC 发布时间戳，能彻底解决 P1.1 的执行时间不一致问题，直接服务于 Clean Zone 的时间点干净性。这是最值得做的单项升级。

**Point-in-Time 宏观数据（替换 FRED 原始接口）**

FRED 只提供最新修订值，无法查询"当时发布的原始数字"。Bloomberg / Haver Analytics 提供 point-in-time 历史宏观数据，能彻底解决 P1.2 的发布滞后假设问题。对 Clean Zone 的数据干净性有直接影响。

### 9.3 升级了但价值有限的方向

| 升级方向 | 原因 |
|---|---|
| 叠加更多免费新闻源 | 问题是质量和时间戳，不是数量，叠加只放大噪音 |
| 引入更高频量化数据（日内/周频） | 系统是季度频率宏观框架，高频数据无增量价值反增噪音 |
| 另类数据（卫星、信用卡消费等） | 成本极高，需专门特征工程，当前阶段性价比极低 |

### 9.4 升级触发条件

Clean Zone 胜率数据出现后，根据诊断结果决定升级方向：

- 若胜率低且辩论逻辑质量高 → 数据源质量是瓶颈，优先升级新闻和宏观数据
- 若胜率低且辩论存在系统性偏差 → 数据源不是问题，优先改进辩论机制
- 若胜率达标 → 数据源维持现状，资源投入模型迭代

## 十、学习表路径架构矛盾 · 待决策项

### 10.1 发现的问题

定位重塑 + 回测写回隔离完成后，系统存在一个深层架构矛盾：**学习表路径在现有架构下已永久失效。**

写回学习表（`QuantPatternLog`、`SkillLibrary`、`LearningLog`）需要同时满足两个条件：

```
is_backtest = False          （七、新增的回测污染门控）
decision_date < 2023-01-01   （原有 TRAIN_TEST_CUTOFF 门控）
```

但所有 live 决策的 `decision_date` 均为发出时间（2025 年之后），永远无法通过 `TRAIN_TEST_CUTOFF` 门控。回测决策虽然 `decision_date` 可以是 2023 年之前，但已被 `is_backtest` 门控拦截。两个条件无法同时满足，学习表永远不会收到任何数据。

### 10.2 当前系统实际依赖的两件事

学习表路径失效后，系统真正运行依赖的只有：

1. **LLM 预训练知识**：模型本身对宏观金融规律的内化
2. **情景记忆注入**：`DecisionLog` → `get_historical_context()` → hist_ctx → 辩论系统参考

这与系统的新定位（规范判断过程、积累决策脉络）完全自洽，学习表路径是为已被否定的旧目标（历史回测写回学习）而设计的遗留架构。

### 10.3 后续两个选项（待决策）

**选项 A：以 Clean Zone 验证通过的 live 决策作为新学习来源**

调整 `TRAIN_TEST_CUTOFF` 向前移动至当前日期附近，或单独为 Clean Zone 高质量决策（胜率达标 + LCS 通过）开一个受控写回通道。这是架构上最干净的学习数据来源——无前瞻污染、无 LLM 历史记忆干扰。

前提条件：Clean Zone 需积累足够样本（建议 ≥ 50 条），且胜率稳定后再考虑，防止在噪音上过拟合。

**选项 B：放弃学习表，接受系统永远只依赖情景记忆**

直接承认 `QuantPatternLog`、`SkillLibrary`、`LearningLog` 在当前定位下无实际用途，将其降级为历史遗留表，不再维护。更简单、更诚实，完全符合"外部纪律约束框架"的定位。

### 10.4 当前决定

暂不修改。学习表保留但休眠，不影响系统正常运行。等 Clean Zone 数据积累后根据实际胜率表现再做选择。

## 十一、Clean Zone 启动前审计 · 待修复问题与可精简内容

### 11.1 背景

在开始积累真实 Clean Zone 样本之前，对系统进行了完整审计，识别出会直接污染样本质量的致命漏洞、影响可靠性的重要问题，以及在新定位下可以去除的冗余内容。**以下问题尚未修复，待处理。**

---

### 11.2 致命漏洞（必须在 Clean Zone 积累前修复）

**`extract_direction` 关键词优先级冲突**

**文件**：`engine/memory.py`

当前实现是顺序扫描关键词列表，谁先被扫到就记谁。当 LLM 输出含有多个方向关键词的文本时（如"从低配调整为标配"同时包含"低配"和"标配"），最终记录的方向取决于关键词在列表中的顺序，而非文本的实际语义。

**影响**：方向字段记录错误 → 三重壁垒验证评分方向反转 → Clean Zone 所有受影响样本的 `accuracy_score` 失真。这是唯一会静默污染 Clean Zone 数据质量的代码级漏洞。

**修复方案**：两轮提取策略：
1. 优先在结论标记（"最终结论"、"建议方向"、"仲裁结论"等）后 150 字符内提取
2. Fallback 用 `rfind` 取全文最后出现的方向关键词

**状态**：✅ 已修复（`engine/memory.py`）

---

### 11.3 重要问题（影响可靠性，尽快修复）

**`decision_date` 未显式设置**

**文件**：`ui/tabs.py` → `save_decision` 调用处

Live 分析调用 `save_decision` 时 `decision_date` 参数为 `None`，由数据库用 `created_at` 时间自动填充。若用户在 UTC 午夜前后跨天运行分析，`created_at` 可能比实际市场日期晚一天，影响 Clean Zone 边界判断（2025-04-01 附近的记录可能被错误分类）。

**修复方案**：在 `_run_sector_analysis` 中显式捕获 `_decision_date = datetime.now(timezone.utc).date()`，传入 `save_decision`，同时复用为 `compute_state_vector` 的日期参数，确保全链路使用同一 UTC 日期基准。

**状态**：✅ 已修复（`ui/tabs.py`）

**`sensitivity_flag` 死字段**

**文件**：`ui/tabs.py`、`engine/memory.py`

`save_decision` 接收并写入 `sensitivity_flag`，但对应的 `run_sensitivity_test()` 函数从未被定义（整个代码库无此函数）。每次 `_run_sector_analysis` 调用都会触发 `NameError`，被外层 `try/except` 静默吞掉，字段始终写入空值。

**影响**：当前不影响运行，但字段存在会误导未来的数据分析。

**修复方案**：从 `_run_sector_analysis` 中移除 `run_sensitivity_test()` 调用和 `sensitivity_flag` 传参。`save_decision` 签名保留参数默认空值，等待真正实现后再启用。

**状态**：✅ 已修复（`ui/tabs.py`）

---

| 内容 | 位置 | 原因 | 建议 |
|---|---|---|---|
| Stress Test Tab | `pages/admin.py` + `engine/scenarios.py` | 压力测试场景在"规范判断过程"定位下无角色，属旧目标遗留 | 可去除 |

## 十二、StressTestLog · 压力情景复盘数据库

**文件**：`engine/memory.py`、`pages/admin.py`

### 背景

Stress Test Tab 原先的分析结果仅存于 `session_state`，关掉页面即消失。经评估，压力情景分析对宏观研究者有复盘价值（"哪些情景我跑过、AI 给了什么判断、与实际结果如何对比"），值得持久化保存。

### 新增内容

**`StressTestLog` 表**（`engine/memory.py`）

轻量设计，只存关键元数据，不存 AI 全文：

| 字段 | 内容 |
|---|---|

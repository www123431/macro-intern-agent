## 二、修复记录

### Fix ①：LCS 验证改用 key pool（pages/admin.py）

**问题**：`verify_pending_decisions()` 被调用时，模型来自 `st.secrets.get("GEMINI_API_KEY")`。若该 secret 未配置或已失效，`model=None`，LCS 静默跳过。所有 Clean Zone 记录的 `lcs_passed` 永远为 `NULL`，LCS 通过率指标完全失效。

**修复**：改为从 key pool 获取模型：

```python
# Before
_vkey   = st.secrets.get("GEMINI_API_KEY") or st.secrets.get("GEMINI_KEY", "")
_vmodel = _make_model(_vkey)

# After
from engine.key_pool import get_pool as _get_pool
_vmodel = _get_pool().get_model()
```

**影响**：LCS 现在与主分析使用同一个 key pool，具备 rotation 和 billing protection。

---

### Fix ②：Scanner 加 XAI 归因块（ui/tabs.py）

**问题**：`generate_ai_reasons()` prompt 是简单的4点摘要，无 `超配/标配/低配` 明确指令，无 `[XAI_ATTRIBUTION]` 块。结果：
- `direction` 提取不可靠
- `confidence_score = NULL`，Brier Score 永远无法计算
- Scanner 决策在 Clean Zone 中形同残缺记录

**修复**：在 prompt 末尾追加 XAI 归因块要求：

```
### [XAI_ATTRIBUTION]
overall_confidence: [0-100]
direction: [超配 / 标配 / 低配]
invalidation_conditions: [1条失效条件]
### [/XAI_ATTRIBUTION]
```

并在 `save_decision()` 调用前解析 `overall_confidence` 写入 `confidence_score`：

```python
_scanner_conf = None
_xai_conf = re.search(r"overall_confidence:\s*(\d+)", analysis)
if _xai_conf:
    _scanner_conf = int(_xai_conf.group(1))
save_decision(..., confidence_score=_scanner_conf, ...)
```

**影响**：Scanner 决策现在有完整 XAI 归因，可参与 Brier Score 计算和置信度校准。

### Fix ③：Macro 决策移除 SPY ticker（ui/tabs.py）

**问题**：Macro 分析是宏观制度性评论（risk-on/off 判断），但 `save_decision()` 传入 `ticker="SPY"`，导致 `verify_pending_decisions()` 用 SPY 的90天收益率对宏观分析打分。语义完全不匹配——宏观分析说"避险"，SPY 涨了，被记为 wrong，但两者根本不是同一个问题。

**修复**：移除 macro `save_decision()` 中的 `ticker="SPY"`：

# After

**影响**：Macro 决策继续存入 DB 供 `get_historical_context()` 调用，但不进入 Triple-Barrier 验证队列，避免污染 Clean Zone 准确率指标。

## 六、补充修复 — Audit confidence_score 空值处理（ui/tabs.py）

**问题**：Audit tab 的 `save_decision()` 调用中：

```python
confidence_score=int(q.get("confidence_score") or 0)
```

当 `confidence_score` 为 `None` 时，`or 0` 将其映射为 `0`。`0` 会被 `_zone_stats()` 中的 Brier Score 公式当作"0% 置信度"参与计算，实际效果是在没有任何置信度信息的情况下人为拉高 Brier Score，污染校准指标。

**修复**：

```python
# Before
confidence_score=int(q.get("confidence_score") or 0)

# After
confidence_score=(int(q["confidence_score"]) if q.get("confidence_score") is not None else None)
```

`None` 会被 `_zone_stats()` 正确过滤，只有真实记录了置信度的决策才进入 Brier Score 计算。

### A. 金融/学术层面不合理（4项）

#### A1 ⚠️ P-hacking 检验使用硬编码常数——幻象统计指标 [最严重]

**位置**：`engine/agent.py:143`

```python
_, p_noise = StrategyAuditor.check_optimizer_curse(0.05, 0.045, 100)
```

三个参数（val_score=0.05, test_score=0.045, n_trials=100）全部硬编码，与当前资产组合数据无关。可数学验证：

```
luck_factor = log1p(100) × 0.01 ≈ 0.0462
gap         = max(0, 0.05 - 0.045) = 0.005
p_noise     = 1 - exp(-0.005 / 0.0462) ≈ 10.3%（常数）
```

结果：Red Team prompt 中每次显示相同的 "P-hacking 风险 10.3%，低于 30% 阈值"——这是一个**对用户撒谎的幻象指标**，声称做了统计检验，实际上没有。`research_node` 中基于此计算的 `confidence_score` 同理为伪指标。

**影响范围**：Audit Tab 的每一次分析都显示这个常数，Clean Zone 记录的 `confidence_score`（来自 research_node）不可信。

**需要做的**：移除 `check_optimizer_curse` 调用或传入实际的 val/test 分数；在 Clean Zone 阶段积累足够样本前，`p_noise` 在 UI 上应标注为"尚无有效估算"。

---

#### A2 ⚠️ 信用利差代理在金融概念上错误

**位置**：`engine/quant.py:268`

```python
cs = float(hyg.pct_change(20).iloc[-1] - ief.pct_change(20).iloc[-1])
```

HYG 与 IEF 的 20 日**价格收益差**不等于信用利差。信用利差（OAS，期权调整利差）在 yield space 以基点计量，需要从债券价格反推收益率。HYG 久期约4年、IEF 约7年，价格对利率敏感度不同，价格差主要反映的是利率效应（久期差异）而非信用风险。

**实际效果**：该指标作为方向性参考仍有一定价值（HYG 比 IEF 跑赢确实隐含风险偏好改善），但不可量化为基点，UI 标注"信用利差(HYG-IEF 20日价差)"是金融误导。

**建议**：将标注改为"风险偏好指标（HYG vs IEF 相对强弱）"，或从 FRED 拉取真实 OAS 数据（`BAMLH0A0HYM2`）。

---

#### A3 ⚠️ 动态 VaR 的 alpha 调整方向相反

**位置**：`engine/quant.py:102`

```python
alpha = 0.05 * (1 + (vix - 20) / 40) if vix > 20 else 0.05
```

VIX=40 时，alpha=0.075，计算第7.5百分位（比第5百分位更接近0），VaR 绝对值**变小**。高波动期风险度量反而更宽松——逻辑方向完全颠倒。

正确逻辑：高 VIX 应用更低 alpha（如 0.01-0.03），使 VaR 更保守。当前公式在压力市场会系统性低估尾部风险。

**建议**：改为 `alpha = 0.05 / (1 + (vix - 20) / 40) if vix > 20 else 0.05`，或直接弃用动态 alpha，改用 GARCH/EWMA 条件波动率。

---

#### A4 🚨 随机8%强制拦截污染 Clean Zone 记录 [需立即修复]

**位置**：`engine/agent.py:286-290`

```python
_RANDOM_AUDIT_RATE = 0.08
if is_robust and random.random() < _RANDOM_AUDIT_RATE:
    is_robust = False  # 强制走 reflection 路径
```

本意是"防止模型过度自信的系统性盲点"，但有根本性实验设计缺陷：

1. **污染记录**：被随机触发的拦截与真实拦截在 DecisionLog 中完全无法区分，该字段未被写入
2. **无法事后清洗**：Clean Zone 积累的任何被拦截记录都可能是随机产生的，不可区分
3. **实验完整性破坏**：向测试集注入了一个不可观测的随机处理变量，违反基本实验设计原则
4. **方向性问题**：Audit Tab 走 reflection 路径的决策（has no ticker）不进入 Triple-Barrier 验证，但仍进入 DecisionLog——这些随机拦截的决策变成了永久 unverified 记录

**需要做的**：在 Clean Zone 开始前移除此代码，或至少在 DecisionLog 中增加 `random_audit_flag` 字段以便事后过滤。

### C. Clean Zone 积累阶段需要解决（3项）

#### C1 🚨 Macro 决策永久无法验证 [需立即有政策决定]

**问题**：`verify_pending_decisions()` 在 `if not ticker: continue` 处跳过 macro 决策。Macro 分析没有对应 ETF ticker，无法经过 Triple-Barrier 验证，所有 macro 记录在 DB 中永久为 `verified=False`，Invalidation Monitor 会持续将它们显示为"逾期"，且数量会随时间线性增长。

**两种解决方案**（需决策）：

- **方案 A（接受不可验证）**：在 `save_decision(tab_type="macro")` 时直接设 `verified=True`，`accuracy_score=None`，并在 DecisionLog 中增加 `skip_verification_reason="macro_no_ticker"`。Macro 记录进入历史上下文但不进入 Clean Zone 统计。Monitor 中过滤掉 macro。
- **方案 B（宏观制度验证）**：用 VIX/SPY 方向性指标对 macro 判断打分（如 risk-on 判断 → 检验 SPY 是否在90天内上涨），接受此方法学缺陷但获得可量化的验证结果。

#### C2 ⚠️ Scanner 决策生成时无实时质量门控

Sector 分析有 Debate Engine 作为生成期质量控制（Blue → Red → Arbitration），Scanner 没有辩论，LCS 只在45-90天后的验证期触发。低质量 Scanner 决策会无条件入库。

**当前阶段**：可接受，但在解读 Clean Zone 结果时应注意：Scanner 的 Clean Zone 样本质量可能系统性低于 Sector，需分 tab_type 解读胜率，不可混合。

#### C3 ⚠️ `horizon` 字段解析脆弱

**位置**：`engine/debate.py:139`

```python
result["horizon"] = "半年(6个月)" if ("半" in raw or "长" in raw) else "季度(3个月)"
```

任何包含"长"字的 AI 回答（如"分析时间较长"、"趋势较长"）都会被错误归类为半年，影响 Triple-Barrier 时间障碍（90天 vs 180天），进而影响 accuracy_score。

**建议**：改为完整短语匹配，如 `"半年" in raw or "6个月" in raw`，避免单字触发误判。

### D. 优先级排序

| 优先级 | 项目 | 理由 |
|--------|------|------|
| 立即 | A4 随机拦截 | 直接污染 Clean Zone 记录，无法事后清洗 |
| 立即 | C1 Macro 无法验证 | 需要政策决定，影响所有 macro 决策的最终状态 |
| 积累阶段前 | C3 horizon 解析 | 影响 accuracy_score 计算准确性 |
| 积累阶段内 | A1 幻象 p_noise | 用户可见的误导性指标，但不污染 Clean Zone |
| 积累后优化 | A2 信用利差标注 | 标注误导，但信号方向有一定参考价值 |
| 积累后优化 | A3 VaR 方向 | 影响 Quant Audit 显示，但不进入 Clean Zone 统计 |
| 技术债 | B1/B2/B3 | 冗余但不影响正确性，可在下个重构周期清理 |

## 十二、制度一致性架构改动

### 背景

高波动市场下，90/180 天预测窗口内宏观制度可能完全翻转。没有制度漂移记录的情况下，`accuracy_score` 无法区分"分析方向错误"与"制度改变导致的 timing 失败"，前 n 条 Clean Zone 样本存在系统性单制度偏差风险。

### 改动一：`regime_drifted` 字段（`engine/memory.py`）

**新增 DecisionLog 字段**：
- `regime_at_verify: String(50)` — 障碍触发日期的 VIX 制度标签
- `regime_drifted: Boolean` — `macro_regime`（决策时）≠ `regime_at_verify` 时为 True

**实现**：`verify_pending_decisions()` 在写入 `verified=True` 后，额外下载 `^VIX` 历史数据，取障碍触发日期附近的 VIX，用 `_regime_from_vix()` 映射为制度标签，与原始 `macro_regime` 比较。VIX 获取失败时字段保持 NULL（非关键路径，不影响主流程）。

**价值**：事后可分层统计——制度稳定样本 vs 制度漂移样本的胜率差异，本身是有研究价值的发现。

### 改动二：λ 复合门控（`engine/memory.py`）

**旧**：`_LASSO_GATE_THRESHOLD = 15`（单一数量门控）

**新**：
```python
_LASSO_GATE_MIN_N       = 20  # 数量门控（提升）
_LASSO_GATE_MIN_REGIMES = 2   # 制度多样性门控（新增）
```

`get_lasso_lambda()` 现在同时检查两个条件：
- Clean Zone 验证样本 ≥ 20
- 样本中覆盖的不同 `macro_regime` ≥ 2

**理由**：单一制度下校准的参数无跨制度泛化能力。如果前20条样本全在高波动/危机制度下产生，用它们解锁 λ 校准毫无意义。两个条件必须同时满足。

`_LASSO_GATE_THRESHOLD` 保留为向后兼容别名（= `_LASSO_GATE_MIN_N`）。

### 改动三：Watchlist 制度漂移标记（`ui/tabs.py`）

当 `regime_drifted=True` 时，决策卡片标题行显示**橙色「制度已漂移」徽章**，提示用户该决策的验证结果需要额外解读——不一定是分析失败，可能是制度变化导致的 timing 问题。

### 改动四：Admin Parameter Status 更新（`pages/admin.py`）

λ 门控说明从"n≥15"更新为"n≥20 且 制度多样性≥2（两项同时满足）"，进度条保留数量进度，文字说明补充制度多样性要求。

### 文件变更

| 文件 | 变更 |
|------|------|
| `engine/memory.py` | DecisionLog 新增 `regime_at_verify` + `regime_drifted` 字段；新增 `_regime_from_vix()` helper；`verify_pending_decisions()` 写入制度漂移；λ 门控改为复合条件；`get_pending_decisions_for_monitor()` 返回新字段 |
| `ui/tabs.py` | Watchlist 卡片新增制度漂移橙色徽章 |
| `pages/admin.py` | λ 门控展示更新为复合条件；更新导入名称 |

## 十三、循环验证风险缓解

### 背景

当前验证链存在认识论循环：AI 生成决策 → AI（LCS）评估逻辑一致性 → AI 系统记录结果。LCS 只验证"AI 自身推理的内部自洽性"，不验证"判断的预测有效性"。真正独立的验证来源只有 Triple-Barrier（使用真实市场价格，AI 无法干预）。

**分析**：系统中循环的实际严重程度不均匀——Triple-Barrier 已经是完全独立的验证；LCS 只在"准入门控"层面存在循环，不影响价格验证结果本身。改进方向是：强化 LCS 的纯门控定位，并引入人工独立信号打破循环。

### 改动一：LCS 纯门控原则显式化（`engine/memory.py`）

**问题**：`run_meta_agent_analysis()` 是系统中唯一没有对 `lcs_passed` 做过滤的学习路径。尽管该函数仅作用于训练集（decision_date < TRAIN_TEST_CUTOFF），理论上影响有限，但逻辑上违反了"LCS 失败 → 不进入任何学习表"的统一原则。

**修复**：在 `run_meta_agent_analysis()` 查询中补充过滤条件：

```python
(DecisionLog.lcs_passed != False) | DecisionLog.lcs_passed.is_(None)
```

**架构原则**（在 docstring 中明确记录）：

> LCS 是纯门控机制，在整个系统中不作为加权因子。  
> 所有学习权重（accuracy_rate、avg_accuracy 等）完全来源于 Triple-Barrier 的实证价格结果。  
> LCS 决定一条记录是否进入学习表，但不扭曲学习表内部的权重。  
> 循环的影响被隔离在准入门控层，不蔓延到权重计算。

### 改动二：人工预验证标注（方案C）

**机制**：在 Watchlist（`render_tab7`）的每张待验证决策卡片下方，新增一个下拉选择器，供用户在**结果出来之前**对决策质量进行独立打分。

**标注选项**：
- `pre_strong` — 逻辑清晰
- `pre_uncertain` — 有疑虑
- `pre_poor` — 明显缺陷

标注值写入 `DecisionLog.human_label` 字段（已有字段，原仅用于 `black_swan`/`analysis_error`）。

**价值**：产生完全独立于 AI 的信号。等 Triple-Barrier 结果积累后，可分析：
- `human_label=pre_strong` 的决策，Triple-Barrier 胜率是否显著高于 `lcs_passed=True`？
- 两者哪个对最终结果的预测力更强？

这是唯一能回答"LCS 是否真的有信息量"的方法，也是打破 AI 循环验证的关键路径。

**`set_human_label()` 扩展**：原函数仅接受 `"black_swan"` 和 `"analysis_error"`，现扩展支持三个预验证标签，并区分行为：
- 预验证标签不清空 `needs_review` 标志（这只影响异常后复核流程）
- 预验证标签可通过"（未标注）"选项被清空（写回 NULL）

**UI**：卡片标题行显示紫色「人工预评: XXX」徽章（已标注时），卡片底部右侧为 3:1 宽度布局的下拉选择器。

### 改动三：DB migration 补丁（`engine/memory.py`）

`_migrate_db()` 的 `dl_extra_columns` 列表中补充了 `regime_at_verify` 和 `regime_drifted` 两列，确保现有 SQLite 数据库在启动时自动执行 `ALTER TABLE`，不再出现 `OperationalError: no such column`。

### 文件变更

| 文件 | 变更 |
|------|------|
| `engine/memory.py` | `run_meta_agent_analysis()` 补充 LCS 门控；`set_human_label()` 扩展支持预验证标签；`get_pending_decisions_for_monitor()` 返回 `human_label`；`_migrate_db()` 补充 `regime_at_verify`/`regime_drifted` 迁移；LCS 纯门控架构原则写入 docstring |
| `ui/tabs.py` | Watchlist 卡片新增预验证标注下拉选择器；导入 `set_human_label` |

## 十六、XAI 块解析规范化（C3 修复）

### 背景

`debate.py` 的 `_parse_xai()` 使用正则 `key: value` 解析 XAI 结构化字段，存在两个问题：

1. **C3 horizon 误判**：`"半" in raw or "长" in raw` 的单字符匹配导致包含"较长"、"趋势长期"等词语的输出被错误分类为"半年"，影响 Triple-Barrier 时间障碍选择
2. **架构层面**：依赖 LLM 输出格式稳定是工程债，任何 prompt 变动或模型升级都可能导致正则静默失效

### 改动

**`engine/debate.py`** 新增三个函数，替换原有 `_parse_xai()`：

| 函数 | 职责 |
|------|------|
| `_parse_xai_json(block)` | 主解析路径：`json.loads()` + schema 验证，horizon 精确匹配 |
| `_parse_xai_regex(block)` | 原有逻辑重命名为 fallback，同时修复 C3（改用 `"半年" in raw`） |
| `_parse_xai(text)` | 编排层：提取 XAI 块 → 先 JSON → 失败则 regex，并记录 debug 日志 |

**`_xai_block_template()`** 同步更新：XAI 块内容从 `key: value` 格式改为完整 JSON 对象，`horizon` 字段明确标注"必须是以下精确字符串之一"。

### 设计原则

- **零破坏风险**：JSON 解析失败时静默降级到 regex，不影响现有数据完整性
- **可观测**：JSON 解析失败时写入 `logger.debug`，可通过日志监控迁移进度
- **horizon C3 同步修复**：regex fallback 路径也从 `"长" in raw` 改为 `"半年" in raw`，彻底消除误判
- **规范化方向**：为后续引入 Gemini `response_schema`（模型层强制 JSON）预留接口
